01

Four features that all react to scroll

They sound alike, but they answer different questions. Get the mental model right and the syntax falls into place.

FeatureThe question it answersCore APIReverses on scroll up?
Scroll-driven“Where is the scrollbar right now?” — progress is tied to scroll position.animation-timelineYes — it’s a scrubber
Scroll-triggered“Has this element crossed a threshold?” — then play once, on its own clock.IntersectionObserverNo — fire & forget
Scroll states“Is this element stuck / snapped / scrollable?” — style based on that state.scroll-state() queriesStyle toggles with state
View transitions“The DOM changed — animate between old and new.” Scroll optional.startViewTransition()n/a — between states
▎i

Quick gut check: scroll-driven = a scrubber you control by scrolling. scroll-triggered = a tripwire that plays an animation. scroll-state = conditional CSS for sticky/snap/scrollable. view transitions = morph between two DOM states (often on navigation, not scroll at all).

Everything below works with no build step. The vanilla examples are copy-paste CSS; the Tailwind examples use the Play CDN — a single <script> tag, covered next.

02

Tailwind with zero install (the CDN)

None of these features have dedicated Tailwind utilities — so the trick is Tailwind’s arbitrary properties: write any CSS declaration inside square brackets and it becomes a class. The Play CDN compiles it in the browser.

⌘HTML — drop this in <head>
<!-- One tag. No npm, no build. (Play CDN = dev/prototyping) -->
<script src="https://cdn.tailwindcss.com"></script>

<!-- optional: register custom keyframes so animate-* can use them -->
<script>
  tailwind.config = {
    theme: { extend: {
      keyframes: { reveal: { from:{opacity:'0',transform:'translateY(40px)'}, to:{opacity:'1',transform:'none'} } },
      animation: { reveal: 'reveal linear both' }
    }}
  }
</script>

The square-bracket pattern you’ll reuse everywhere

Inside arbitrary values, spaces become underscores. So scroll(root block) is written scroll(root_block).

twany property → a class
<!-- property:value with spaces written as underscores -->
<div class="[animation-timeline:scroll(root_block)] origin-left"></div>

<!-- compiles to → div { animation-timeline: scroll(root block); transform-origin: left } -->
▎!

The CDN is for prototyping. It compiles on every page load and ships the whole engine to the browser. Perfect for these demos and learning; for production, use the Tailwind CLI/PostCSS build. Either way — no package is installed on your machine for the CDN.

One more honest caveat: at-rules and pseudo-elements (@container scroll-state(), @view-transition, ::view-transition-old()) can’t be expressed as utility classes. For those, Tailwind users still drop a small <style> block — shown in each section below.

03

Scroll-driven animations

A normal CSS animation, but its timeline is the scrollbar instead of the clock. Scroll forward → it advances; scroll back → it rewinds; stop → it freezes. You’re scrubbing it.

Two anonymous timelines do most of the work:

  • scroll() — progress of a scroll container (e.g. the page). scroll(root block) = vertical page scroll, 0% at top → 100% at bottom.
  • view() — progress of this element moving through the scrollport. Great for “animate in as it enters.”

Also live on this page: the thin progress bar pinned to the very top is a one-liner scroll(root block) animation.

Plain CSS vanilla

cssscroll-driven.css
/* 1 · page reading-progress bar — tied to root scroll */
@keyframes grow { from{transform:scaleX(0)} to{transform:scaleX(1)} }
.progress {
  transform-origin: left;
  animation: grow linear both;
  animation-timeline: scroll(root block);   /* the whole-page scroller */
}

/* 2 · reveal a card as IT enters the viewport */
@keyframes reveal {
  from { opacity:0; transform:translateY(40px) }
  to   { opacity:1; transform:none }
}
.card {
  animation: reveal linear both;
  animation-timeline: view();            /* element-relative timeline */
  animation-range: entry 5% cover 38%;    /* start/finish window */
}

The scrubber demo above uses a named timeline so the scroll box can drive a sibling ring — plus @property to make a conic-gradient angle animatable:

cssscrubber.css
/* make an angle interpolable so the ring can fill */
@property --deg { syntax:'<angle>'; inherits:false; initial-value:0deg }
@keyframes fill { from{--deg:0deg} to{--deg:360deg} }

/* common ancestor exposes the name to siblings */
.wrap     { timeline-scope: --box }
.scroller { overflow-y:auto; scroll-timeline: --box block }   /* declares it */

.ring {
  background: conic-gradient(#7c9cff var(--deg), #ffffff12 0);
  animation: fill linear both;
  animation-timeline: --box;     /* references it, even as a sibling */
}

Without timeline-scope a named timeline is only visible to the scroller’s descendants; with it, any element under .wrap can use it.

Tailwind (CDN) arbitrary props

twscroll-driven.html
<!-- progress bar (keyframes 'grow' registered in tailwind.config) -->
<div class="fixed top-0 left-0 h-[3px] w-full origin-left
            animate-[grow_linear_both] [animation-timeline:scroll(root_block)]"></div>

<!-- reveal-on-enter card (keyframes 'reveal' registered in config) -->
<div class="rounded-xl p-5 bg-white/5
            animate-reveal
            [animation-timeline:view()]
            [animation-range:entry_5%_cover_38%]"> … </div>
▎i

Always add both (fill mode) and omit any duration — with a scroll timeline the animation maps 0→100% across the range, so a time-based duration is meaningless.

04

Scroll-triggered animations

The opposite reflex: cross a threshold once, then the animation plays on its own timing and doesn’t rewind when you scroll back. A tripwire, not a scrubber.

There’s an experimental pure-CSS animation-trigger on the horizon, but the reliable, ship-it-today tool is IntersectionObserver toggling a class. The class carries an ordinary time-based animation.

CSS + a few lines of JS works everywhere

csstrigger.css
.badge        { opacity:0; transform:translateY(18px) scale(.8) }
.badge.fired  { animation: pop .7s cubic-bezier(.2,1.4,.4,1) forwards }
@keyframes pop { to{ opacity:1; transform:none } }
jstrigger.js
const io = new IntersectionObserver((entries) => {
  for (const e of entries) {
    if (e.isIntersecting) {
      e.target.classList.add('fired');   // play once…
      io.unobserve(e.target);            // …then stop watching
    }
  }
}, { threshold: 0.6 });             // the activation range

document.querySelectorAll('.badge').forEach(el => io.observe(el));

Tailwind (CDN) utilities + tiny JS

twtrigger.html
<!-- start hidden via utilities; JS swaps the state classes -->
<div data-trigger
     class="opacity-0 translate-y-5 transition-all duration-700 ease-out">↑ 98%</div>

<script>
  const io = new IntersectionObserver((es) => es.forEach(e => {
    if (!e.isIntersecting) return;
    e.target.classList.remove('opacity-0','translate-y-5');  // → fades up, once
    io.unobserve(e.target);
  }), { threshold: 0.6 });
  document.querySelectorAll('[data-trigger]').forEach(el => io.observe(el));
</script>
▎!

Don’t reach for scroll-driven here. If the effect should keep replaying every time you scroll past, that’s scroll-driven. If it should fire once and stay, it’s scroll-triggered — and unobserve() is what makes it one-shot.

05

Scroll-state container queries

Not an animation at all — a conditional. You ask a container about its scroll state and apply styles when the answer changes. Three questions you can ask:

  • stuck — is a position:sticky element currently pinned to an edge? (top, bottom, …)
  • snapped — is this child snapped by scroll-snap? (x, y, block, inline)
  • scrollable — can the container still scroll in a direction? (great for “more ↓” affordances)

The header above declares container-type: scroll-state and is position: sticky. The moment it pins to the top, the @container scroll-state(stuck: top) rule paints it with the gradient, adds a shadow, and tightens the letter-spacing.

Plain CSS vanilla

cssscroll-state.css
/* the scroll container becomes a scroll-state query container */
.box {
  overflow: auto;
  container-type: scroll-state;
}
.box .bar {
  position: sticky; top: 0;
  transition: background .25s, box-shadow .25s;
}

/* when .bar is pinned to the top edge → restyle it */
@container scroll-state(stuck: top) {
  .bar {
    background: linear-gradient(90deg, #7c9cff, #5fe3d0);
    box-shadow: 0 10px 24px -10px rgb(124 156 255 / .8);
  }
}

The carousel demo uses the snapped query instead — put the container on each slide:

csssnapped.css
.row   { display:flex; overflow-x:auto; scroll-snap-type:x mandatory }
.slide { scroll-snap-align:center; container-type:scroll-state }
.card  { transform:scale(.9); opacity:.45; transition:.3s }

/* is THIS slide the one currently snapped? then spotlight it */
@container scroll-state(snapped: inline) {
  .card { transform:scale(1); opacity:1; border-color:#7c9cff }
}

Tailwind (CDN) utility + small style block

The container-type is a utility; the @container scroll-state() query has no class form, so it lives in a <style> tag.

twscroll-state.html
<div class="h-56 overflow-auto rounded-xl [container-type:scroll-state]">
  <div class="bar sticky top-0 p-3 transition-all">Sticky header</div>
  <div class="h-[600px]">…tall content…</div>
</div>

<style>
  @container scroll-state(stuck: top) {
    .bar { background:linear-gradient(90deg,#7c9cff,#5fe3d0); color:#0a0c15 }
  }
</style>
▎i

Newest of the bunch. scroll-state() queries are Chromium-first (2025). Treat the restyle as progressive enhancement — without support, the header simply keeps its default look.

06

View transitions

The odd one out: it isn’t about scroll at all. You change the DOM, and the browser animates between the old and new states — a crossfade by default, or a shared-element morph when you name things.

Two flavors:

  • Same-document — wrap a DOM update in document.startViewTransition(). Perfect for SPA-style swaps, filters, expand/collapse.
  • Cross-document — opt in with one at-rule and the browser animates real page navigations (MPA). Default: a clean crossfade.

Same-document startViewTransition

jsview-transition.js
function expand(tile) {
  const update = () => {
    tile.classList.add('vt-active');   // gives it view-transition-name
    showDetailFor(tile);                // your DOM change
  };
  // progressive enhancement: animate if supported, else just update
  if (document.startViewTransition) document.startViewTransition(update);
  else update();
}
cssview-transition.css
/* matching names on old & new → the browser morphs between them */
.vt-active { view-transition-name: vt-hero }   /* must be unique per snapshot */

/* tune the generated pseudo-elements like any animation */
::view-transition-old(vt-hero),
::view-transition-new(vt-hero) {
  animation-duration: .45s;
  animation-timing-function: cubic-bezier(.2,.7,.2,1);
}

The clip-path wipe demo names the panel, then animates only the new snapshot with a growing circle anchored to the click (JS sets --cx/--cy):

cssclip-wipe.css
.panel { view-transition-name: panel }

::view-transition-old(panel) { animation: none }              /* old sits still… */
::view-transition-new(panel) { animation: wipe .55s both }   /* …new wipes over */

@keyframes wipe {
  from { clip-path: circle(0% at var(--cx) var(--cy)) }
  to   { clip-path: circle(150% at var(--cx) var(--cy)) }
}
jsclip-wipe.js
panel.addEventListener('click', (e) => {
  // anchor the circle to the pointer, as a % of the panel box
  const r = panel.getBoundingClientRect();
  panel.style.setProperty('--cx', (e.clientX - r.left) / r.width  * 100 + '%');
  panel.style.setProperty('--cy', (e.clientY - r.top)  / r.height * 100 + '%');
  document.startViewTransition ? document.startViewTransition(flipTheme) : flipTheme();
});

Cross-document MPA · one at-rule

csson every page you navigate between
/* put this on BOTH the old and new page — same-origin */
@view-transition { navigation: auto }

/* default = crossfade. Name shared elements (e.g. a thumbnail that
   becomes a hero) to morph them across the navigation. */
.thumbnail { view-transition-name: hero-image }

Tailwind (CDN) utility name + style block

twview-transition.html
<!-- the NAME is a utility; the at-rule + pseudos stay in <style> -->
<img class="rounded-xl [view-transition-name:hero-image]">

<style>
  @view-transition { navigation:auto }
  ::view-transition-old(hero-image),
  ::view-transition-new(hero-image) { animation-duration:.4s }
</style>
▎i

Honor user settings: wrap big motion in @media (prefers-reduced-motion: reduce) and shorten or disable the transition. The default crossfade is already gentle.

Support, in one breath

  • Scroll-driven (scroll()/view()) — Chromium 115+, Firefox & Safari rolling in. Degrades to “no animation,” so it’s safe to layer on.
  • Scroll-triggered — IntersectionObserver is universal. The bulletproof choice.
  • Scroll states — Chromium-first (2025), newest here. Pure progressive enhancement.
  • View transitions — same-document broadly in Chromium/Safari; cross-document Chromium-led. Feature-detect document.startViewTransition and fall back to an instant update.
▎✓

Pattern for all four: they’re enhancements. Build the page so it works with none of them, then let supporting browsers add the polish. Always respect prefers-reduced-motion.
