Part I — The format
01

What OKF actually is

The Open Knowledge Format is an open, vendor-neutral spec for representing what an organization knows — the metadata, context, and curated insight around your data and systems — as a directory of Markdown files. It was published by Google Cloud's Knowledge Catalog team in June 2026.

The whole idea is deliberately small: a folder of .md files, each with a YAML frontmatter block, linking to each other with ordinary Markdown links. There is no schema registry, no central authority, and no required tooling. As the spec puts it: if you can cat a file you can read OKF; if you can git clone a repo you can ship it.

  • Readable by humans without tooling — it's just Markdown.
  • Parseable by agents without a bespoke SDK — it's just YAML + Markdown.
  • Diffable in version control — clean line-by-line diffs, blame, PR review.
  • Portable across tools, orgs, and time — ship it as a tarball or a git repo.

The framing in the source video — "the simple folder replacing vector databases" — is the provocation. The reality is more useful: OKF gives an agent a pre-structured graph it can traverse deliberately, instead of a pile of chunks it has to guess through. Known facts (your tables, APIs, metrics, runbooks) live in files; dynamic fuzzy lookups can still use retrieval. More on that split in §08.

▎i

The pattern has a lineage: Andrej Karpathy's "LLM wiki" idea — instead of re-searching raw documents on every query, have the model incrementally build and maintain a living, structured wiki. OKF standardizes the container for that wiki.

02

Why a folder beats a vector database

To give a model context today, you usually dump documents into a vector store and hope similarity search surfaces the right chunk. That approach loses two things OKF keeps:

  • Structure. Chunking shreds the relationships between things. OKF keeps one concept per file and expresses relationships as explicit links — the directory becomes a graph, not a flat list.
  • Verifiability. With retrieval it's hard to know what the model actually saw. With OKF the agent reads named files by path; every claim can cite the exact concept it came from.

And because a bundle is just files, you get properties a service-owned metadata store can't give you for free:

prop / git-native

Versioned like code

Bundles live in git. PRs, diffs, blame, and review just work — knowledge curation becomes normal software engineering.

prop / portable

No lock-in

A bundle is a directory. Tarball it, host it in any repo, mount it from any filesystem. No proprietary API stands between you and your metadata.

prop / composable

Tools already speak it

Obsidian, Notion, MkDocs, Hugo, Jekyll already render Markdown + frontmatter. Your bundle is browsable and editable with zero custom UI.

prop / dual

Structured + prose

Frontmatter holds the few fields you query on; the Markdown body holds the schemas, prose, and example queries that humans and LLMs actually read.

03

Anatomy of a bundle

A Knowledge Bundle is the unit of distribution: a directory tree of Markdown files. The layout is up to you — organize concepts however the knowledge makes sense.

📁acme-knowledge/
# A bundle is just a directory tree of .md files.
acme-knowledge/
├── index.md              # optional: a listing for progressive disclosure
├── log.md                # optional: chronological change history
├── datasets/
│   ├── index.md
│   └── sales.md          # one concept = one file
├── tables/
│   ├── index.md
│   ├── orders.md
│   └── customers.md
└── playbooks/
    └── freshness-alert.md

Two terms carry the whole model:

  • Concept — a single unit of knowledge, represented as one Markdown document. It can describe a tangible asset (a table, an API endpoint), an abstract idea (a metric, a business process), or anything in between.
  • Concept ID — the file's path within the bundle, minus the .md. So tables/orders.md has the ID tables/orders. The path is the identifier.
▎i

Only two filenames are reserved, at any level of the tree: index.md (a directory listing) and log.md (an update history). Every other .md file is a concept. We'll cover both reserved files in §06.

04

The concept document

Every concept is a UTF-8 Markdown file with two parts: a YAML frontmatter block (delimited by --- lines) and a Markdown body. Here's a concept bound to a real resource:

mdtables/orders.md
---
type: BigQuery Table            # REQUIRED — the only mandatory field
title: Customer Orders
description: One row per completed customer order.
resource: https://console.cloud.google.com/bigquery?p=acme&d=sales&t=orders
tags: [sales, orders, revenue]
timestamp: 2026-05-28T14:30:00Z
---

# Schema

| Column        | Type      | Description                              |
|---------------|-----------|------------------------------------------|
| `order_id`    | STRING    | Globally unique order identifier.        |
| `customer_id` | STRING    | FK into [customers](/tables/customers.md). |
| `total_usd`   | NUMERIC   | Order total in US dollars.               |

# Joins

Joined with [customers](/tables/customers.md) on `customer_id`.
Part of the [sales dataset](/datasets/sales.md).

Only type is required. Everything else is recommended-but-optional, and you may add any producer-defined keys you like — conformant consumers preserve unknown fields and never reject a document for having them.

Frontmatter fieldRequired?Purpose
typeYesShort string identifying the kind of concept — BigQuery Table, API Endpoint, Metric, Playbook. Free-form; consumers must tolerate unknown values.
titleNoHuman-readable display name. If omitted, consumers may derive one from the filename.
descriptionNoOne-sentence summary. Used by index generators, search snippets, and previews.
resourceNoA canonical URI for the underlying asset (Grafana, Snowflake, Confluence, a BigQuery console link…). Absent for purely abstract concepts.
tagsNoA YAML list of short strings for cross-cutting categorization.
timestampNoISO 8601 datetime of the last meaningful change.
<anything>NoProducers may add arbitrary keys; consumers preserve them on round-trip.

Bodies are plain Markdown. Favor structural Markdown — headings, tables, fenced code — over freeform prose, since structure helps both human reading and agent retrieval. Three headings carry conventional meaning when they apply: # Schema, # Examples, and # Citations.

05

Links make it a graph, not a tree

The directory hierarchy gives you implicit parent/child structure. Everything richer — foreign keys, joins, "depends on", "see also" — is expressed with ordinary Markdown links. A link from concept A to B simply asserts a relationship; the kind of relationship lives in the surrounding prose, not the link.

mdlink forms
# Absolute (bundle-relative) — recommended. Starts with "/", stable under moves.
See the [customers table](/tables/customers.md) for the join key.

# Relative — standard Markdown relative paths.
See the [neighboring concept](./other.md).

Two consequences worth internalizing:

  • A foreign key, a join, and a "see also" are all the same primitive — a link. That uniformity is what makes the whole bundle traversable by a dumb reader (or a smart agent).
  • Broken links are legal. A link whose target doesn't exist yet isn't malformed — it may just be not-yet-written knowledge. Consumers must tolerate it. This is what lets bundles grow and get refactored incrementally.
▎i

Because links are directed edges, a consumer can compute backlinks ("cited by") by reversing the graph — exactly what the reference visualizer in the OKF repo does to render a force-directed graph of a bundle.

06

index.md & log.md

index.md is the mechanism behind progressive disclosure — the thing that lets an agent (or a human) see what's available before opening individual documents. It may appear in any directory, including the root. It has no frontmatter; its body is sections of links, ideally carrying each concept's description.

mdtables/index.md
# Core tables

* [Orders](orders.md) - one row per completed customer order
* [Customers](customers.md) - one row per customer, with lifetime value

# Related

* [Sales dataset](/datasets/sales.md) - all sales-related tables

log.md is an optional, date-grouped change history for its scope, newest first. It's slightly redundant if you already use git — but it travels with the bundle when git history doesn't.

mdlog.md
# Directory Update Log

## 2026-05-22
* **Update**: Added [Customer Metrics](/tables/customer-metrics.md).
* **Creation**: Established the [Dataplex Playbook](/playbooks/dataplex.md).

## 2026-05-15
* **Initialization**: Created foundational directory structure.

Date headings use ISO 8601 YYYY-MM-DD. The leading bold word (**Update**, **Creation**, **Deprecation**) is a convention, not a requirement. A bundle-root index.md is also the one place frontmatter is permitted in an index — to declare okf_version: "0.1".

07

The three rules (conformance)

The entire spec reduces to three requirements. A bundle is conformant with OKF v0.1 if:

rule / 01

Parseable frontmatter

Every non-reserved .md file contains a parseable YAML frontmatter block.

rule / 02

A non-empty type

Every frontmatter block contains a non-empty type field.

rule / 03

Reserved files behave

index.md and log.md, when present, follow their defined structure.

Everything else is soft guidance. A conformant consumer must not reject a bundle for: missing optional fields, unknown type values, extra frontmatter keys, broken cross-links, or missing index.md files. This permissive model is intentional — it keeps bundles useful as they grow and get partially generated by agents.

Citations

When a concept's body makes claims sourced from external material, list those sources under a # Citations heading at the bottom, numbered. Citation links can be absolute URLs, bundle-relative paths, or paths into a references/ subdirectory that mirrors external material as first-class OKF concepts.

md# Citations
# Citations

[1] [BigQuery dataset announcement](https://cloud.google.com/blog/...)
[2] [Internal data-quality runbook](/references/data-quality.md)
08

OKF vs. RAG — complementary, not rivals

Despite the headline, OKF doesn't kill retrieval. They solve different problems and most serious agent stacks will run both.

DimensionOKFRAG / vector search
shapeA graph you traverse deliberatelyChunks ranked by similarity
retrievalRead named files by path; follow linksFuzzy match at query time
best forKnown facts — schemas, metrics, APIs, runbooks, org structureLong-tail lookups across many unstructured docs
hallucinationLow — exact, citable, no embedding stepCan surface the wrong chunk
updatesEdit a file, commit a diffRe-chunk and re-embed

The heuristic: put what your org knows in OKF for direct, verifiable reads; keep a vector index for fuzzy search over the long tail. Part II builds the OKF half first (§10–§13), then bolts on retrieval for the tail (§14).

09

Build your first bundle

You don't need any tooling. Make a directory, write a concept, add an index. That's a valid OKF bundle.

⌘Terminal
mkdir -p acme-knowledge/tables && cd acme-knowledge

# one concept
cat > tables/orders.md <<'MD'
---
type: BigQuery Table
title: Customer Orders
description: One row per completed customer order.
---
# Schema
| Column | Type | Description |
|--------|------|-------------|
| order_id | STRING | Unique order identifier. |
MD

# a root index for progressive disclosure
cat > index.md <<'MD'
# Tables
* [Orders](tables/orders.md) - one row per completed customer order
MD

git init && git add . && git commit -m "okf: seed bundle"

A tiny conformance check is a dozen lines — read every non-reserved .md, confirm it has frontmatter with a non-empty type:

jsvalidate.mjs
import { readFileSync } from 'node:fs';
import { globSync } from 'node:fs';   // Node 22+: fs.globSync

const reserved = new Set(['index.md', 'log.md']);
let errors = 0;

for (const file of globSync('**/*.md')) {
  if (reserved.has(file.split('/').at(-1))) continue;
  const text = readFileSync(file, 'utf8');
  const fm = text.match(/^---\n([\s\S]*?)\n---/);
  if (!fm || !/^type:\s*\S/m.test(fm[1])) {
    console.error(`✗ ${file}: missing frontmatter or non-empty type`);
    errors++;
  }
}
console.log(errors ? `${errors} error(s)` : '✓ conformant OKF v0.1');
▎i

For production authoring, the OKF repo ships a reference enrichment agent that generates bundles from a BigQuery dataset plus seed docs, and a self-contained HTML visualizer that renders any bundle as an interactive graph. There's also a browser validator at okf.md. But none of it is required — the format is the contribution.

Part II — Ship a chatbot on Next.js (serverless)
10

The 2026 stack

Goal: a chatbot that answers from your OKF bundle, deployed as serverless functions, that stays honest because the model reads named files and cites them. Here's the shape.

layer / app

Next.js App Router

A single Route Handler at app/api/chat/route.ts runs as a serverless function. The UI streams tokens over HTTP.

layer / ai

Vercel AI SDK v5

ai + @ai-sdk/anthropic. streamText handles streaming and the tool-calling loop; useChat handles the client.

layer / model

Claude

claude-sonnet-5 for chat — fast, cheap, 1M context. Swap one string to claude-opus-4-8 for the hardest reasoning.

layer / knowledge

The OKF bundle

Your /okf folder, shipped with the deploy or fetched from object storage. The model traverses it through tools.

⌘Terminal
npx create-next-app@latest okf-chat
cd okf-chat
npm install ai @ai-sdk/anthropic @ai-sdk/react zod

# the model provider reads this automatically
echo 'ANTHROPIC_API_KEY=sk-ant-...' >> .env.local

# drop your bundle in at the repo root
cp -r ../acme-knowledge ./okf
▎i

Why the Vercel AI SDK here? The @ai-sdk/anthropic provider calls the Claude API natively (it is not an OpenAI-compatible shim) and gives you client-side streaming, a built-in tool loop, and provider-level prompt caching — exactly the pieces an OKF chatbot needs. If you prefer, the official @anthropic-ai/sdk works too; the OKF logic is identical.

11

Where the bundle lives at runtime

A serverless function is stateless and cold-starts. You have three sane options for getting the bundle in front of it:

  • In the repo (simplest). Keep /okf in the project and read it with fs at request time. Requires the Node.js runtime (not Edge) and a bundling hint so the files ship with the function.
  • Object storage. Put the bundle in Vercel Blob / S3 / R2 and fetch concepts on demand. Decouples knowledge updates from redeploys; cache aggressively.
  • Build-time import. For small, static bundles, import the Markdown as strings at build. Works on Edge, but every edit needs a rebuild.

This guide uses the in-repo path. On Next.js 15+, tell the tracer to include the folder so it isn't shaken out of the serverless bundle:

jsnext.config.js
/** @type {import('next').NextConfig} */
module.exports = {
  // ship every file under /okf with the chat function
  outputFileTracingIncludes: {
    '/api/chat': ['./okf/**/*'],
  },
};
▎!

OKF is readable by design — never put secrets in it. A bundle is plaintext the model reads and can quote back. Keep API keys, tokens, and PII out of the Markdown; the resource field should point at a system, not embed credentials for it.

12

Retrieval by traversal

Here's the insight that makes OKF click for chatbots: the format already ships the exact primitives an agent needs to navigate it. You don't build a retriever — you expose two tools and let the model do progressive disclosure.

  1. Put the root index.md in the system prompt (cached — see §13). The model always knows the map.
  2. Give it read_concept(id) — open one concept by its path/ID.
  3. Give it list_dir(path) — read a sub-directory's index.md to go one level deeper.

The loop is: read the index → decide which concepts matter → open them → follow their links by calling read_concept again → answer, citing paths. The model traverses the knowledge graph on demand, pulling only the files a given question needs into context — instead of you stuffing the whole bundle in, or guessing chunks. That's the whole retrieval strategy.

a user question
"which table has revenue?"
↓
system prompt · root index.md
the map — sent once, prompt-cached
↓
⟲ traversal loop
Claude picks what it needs
→
read_concept(id) · list_dir(path)
concept body + its links
←
readFile in /okf
↑ follow a link or go deeper → repeat until answered
↓
answer + cited concept paths
e.g. tables/orders — verifiable, no embeddings
▎i

This is why OKF is described as agent-native: concept IDs are stable handles, index.md is a menu, and links are edges. A tool-calling model consumes all three with no adapter — the tools below are barely more than readFile.

13

The chat route

Everything comes together in one Route Handler. It loads the root index once per cold start, caches the stable system prompt with Claude prompt caching, and hands the model the traversal tools.

tsapp/api/chat/route.ts
import { anthropic } from '@ai-sdk/anthropic';
import {
  streamText, tool, convertToModelMessages, stepCountIs, type UIMessage,
} from 'ai';
import { z } from 'zod';
import { readFile } from 'node:fs/promises';
import { join } from 'node:path';

export const runtime = 'nodejs';   // need fs to read the bundle
export const maxDuration = 30;

const BUNDLE = join(process.cwd(), 'okf');

// resolve an OKF concept id / path safely inside the bundle
async function readInBundle(rel: string) {
  const clean = rel.replace(/\\/g, '/').replace(/(^\/|\.\.)/g, '');
  return readFile(join(BUNDLE, clean), 'utf8');
}

// loaded once per cold start — small, stable, cacheable
const rootIndex = await readInBundle('index.md');

export async function POST(req: Request) {
  const { messages }: { messages: UIMessage[] } = await req.json();

  const result = streamText({
    model: anthropic('claude-sonnet-5'),
    stopWhen: stepCountIs(6),          // allow a few traversal hops
    messages: [
      {
        role: 'system',
        content:
          "You answer from the company's OKF knowledge bundle. " +
          'Start from the index below, open the concepts you need with ' +
          'read_concept, follow their links, and cite the concept path ' +
          '(e.g. tables/orders) for every claim.\n\n# index.md\n' + rootIndex,
        // cache the stable system + index → repeat turns read it at ~0.1x
        providerOptions: { anthropic: { cacheControl: { type: 'ephemeral' } } },
      },
      ...convertToModelMessages(messages),
    ],
    tools: {
      read_concept: tool({
        description: 'Read one OKF concept by id (path without .md), e.g. "tables/orders".',
        inputSchema: z.object({ id: z.string() }),
        execute: async ({ id }) =>
          readInBundle(id + '.md')
            .catch(() => `No concept "${id}". Check the index for valid ids.`),
      }),
      list_dir: tool({
        description: 'Read a sub-directory index (progressive disclosure), e.g. "tables".',
        inputSchema: z.object({ path: z.string() }),
        execute: async ({ path }) =>
          readInBundle(join(path, 'index.md'))
            .catch(() => `No index under "${path}".`),
      }),
    },
  });

  return result.toUIMessageStreamResponse();
}

The client is a small useChat component: it POSTs the message list to the route, then walks each message's parts — rendering text bubbles and a badge for every tool call, so you can watch the model open concepts live.

tsxapp/page.tsx
'use client';
import { useChat } from '@ai-sdk/react';
import { useState } from 'react';

export default function Chat() {
  const { messages, sendMessage, status } = useChat();
  const [input, setInput] = useState('');

  return (
    <main className="chat">
      {messages.map((m) => (
        <div key={m.id} data-role={m.role}>
          {m.parts.map((part, i) => {
            // plain assistant / user text → a bubble
            if (part.type === 'text')
              return <p key={i} className="bubble">{part.text}</p>;

            // a tool call: read_concept / list_dir / search_docs
            if (part.type.startsWith('tool-')) {
              const name = part.type.slice(5);
              return (
                <div key={i} className="tool">
                  🔧 {name}({JSON.stringify(part.input)}) · {part.state}
                </div>
              );
            }
            return null;
          })}
        </div>
      ))}

      <form
        onSubmit={(e) => {
          e.preventDefault();
          if (!input.trim()) return;
          sendMessage({ text: input }); // v5: send a text part
          setInput('');
        }}
      >
        <input
          value={input}
          onChange={(e) => setInput(e.target.value)}
          placeholder="Ask about the knowledge base…"
          disabled={status !== 'ready'}
        />
      </form>
    </main>
  );
}
▎i

Prompt caching pays off fast here. The system prompt + root index are byte-identical across turns, so marking them ephemeral lets every follow-up read that prefix at ~0.1× input cost instead of full price. The cached prefix must clear the model's minimum (≈1024 tokens on Sonnet) — a real index easily does. Keep anything volatile (timestamps, per-request IDs) out of that block or you'll silently invalidate the cache.

14

Add RAG for the long tail

OKF is authoritative for structured, known facts. For fuzzy search over a pile of unstructured docs — support tickets, PDFs, wiki pages — add a vector-search tool alongside the traversal tools. The model picks: traverse OKF for "what's the schema of orders?", search for "have we seen this error before?"

tsa third tool
// alongside read_concept / list_dir in the same `tools` object
search_docs: tool({
  description:
    'Fuzzy semantic search over unstructured docs. Use for open-ended ' +
    'lookups; prefer read_concept for known tables, metrics, and runbooks.',
  inputSchema: z.object({ query: z.string(), k: z.number().default(5) }),
  execute: async ({ query, k }) => {
    const hits = await vectorSearch(query, k);  // pgvector / Neon / Upstash…
    return hits.map(h => `# ${h.title}\n${h.text}`).join('\n\n---\n\n');
  },
}),

A tidy trick: mirror anything the tail search keeps surfacing into a references/ subdirectory as first-class OKF concepts. Over time the "known" set grows and the fuzzy set shrinks — the LLM-wiki pattern in action. This is also where the model's own edits could feed back: an agent can write OKF files as it learns, then read them next session.

15

Deploy & keep it fresh

Ship it and treat the bundle like source:

  • Deploy. vercel deploy. The route runs as a Node.js serverless function; the outputFileTracingIncludes hint from §11 carries /okf into it.
  • Keep it current with git. Editing knowledge is a pull request against Markdown — humans or an enrichment agent open PRs, you review the diff, merge, redeploy. That's the entire "content pipeline."
  • Decouple if you need to. For frequent updates without redeploys, move the bundle to Blob/R2 and have the tools fetch concepts (cache them per instance). Same tool signatures — only readInBundle changes.
  • Mind the cache TTL. Claude's ephemeral cache lasts ~5 minutes; bursty chat traffic keeps it warm on its own. The root index loads once per cold start, so cold starts stay cheap.
▎i

What you end up with: a chatbot whose knowledge is a reviewable folder of Markdown, that answers by reading named files and citing their paths, and that degrades gracefully — a missing concept or broken link is a legal state, not a crash. That's the OKF payoff carried all the way to production.

From here: generate bundles from your real catalog with the reference agent, add per-directory log.md files so the model can reason about recency, and let the agent write back learnings as new concepts. The folder grows; the format stays a folder.
