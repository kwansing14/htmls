The TL;DR

Sell businesses an "AI employee" — not an automation, not a SaaS. Charge $5K–$10K/month for unlimited agents, unlimited usage, monitoring, security and ongoing changes. Run everything on cloud computers (Orgo) controlled by a Telegram-driven meta-agent. One person can manage 20+ customers.

Why this works in 2026

Frontier models are now reliable enough at tool-calling that an agent can hold down a real seat in a small business. The bottleneck has shifted from capability to distribution + ops — exactly what a solo operator can own.

1 · Runtime & Orchestration

Orgo — Cloud Computers for Agents
SUBSTRATE
A full Ubuntu/Windows VM per agent, provisioned in <500ms, controllable via screenshot + mouse + keyboard + bash. This is what every worker agent lives inside.

Use case in the business

One Orgo VM = one "AI employee." It has its own desktop, its own browser session, its own logged-in apps (Superhuman, Trello, Granola, Obsidian). The meta-agent spins these up, and each customer gets their own pool.

Install

bash 
pip install orgo
export ORGO_API_KEY=sk_live_xxx

Minimal working example

python 
from orgo import Computer

# spin up a fresh cloud desktop
computer = Computer()

# option A: drive it yourself
computer.bash("sudo apt-get install -y firefox")
computer.left_click(x=500, y=300)
computer.type("hello from the agent")
shot = computer.screenshot()  # PIL image, hand to a vision model

# option B: hand the wheel to a model
computer.prompt("Open Gmail and reply to the latest email from Acme Corp")

computer.shutdown()

Where it slots in

Meta-agent calls: client.computers.create() per new worker.
Worker agent runs: Hermes inside the VM, looping on tools.

Docs: docs.orgo.ai · PyPI: pypi.org/project/orgo

Hermes Agent — The Runtime Inside Each VM
RUNTIME
Open-source autonomous agent from Nous Research with persistent memory, automatic skill creation, and multi-platform reach. Installs into Linux/macOS/WSL2 with one curl.

Use case in the business

Hermes is the loop running inside an Orgo VM. It owns the agent's identity, holds memory between sessions, and exposes a chat/voice/CLI surface. You don't build the loop — you ship skills into it.

Install (run inside the Orgo VM)

bash 
curl -fsSL https://hermes-agent.org/install.sh | bash
hermes init --name "acme-sdr-agent"
hermes start --model gpt-5.5

Adding a custom skill

python 
# ~/.hermes/skills/triage_inbox.py
from hermes import skill, memory

@skill(name="triage_inbox", schedule="*/10 * * * *")
def triage_inbox(ctx):
    emails = ctx.tools.email.list(unread=True)
    for e in emails:
        if "invoice" in e.subject.lower():
            ctx.tools.trello.create_card(list_id=memory.get("ap_list"),
                                         title=e.subject, body=e.body)
            e.archive()

Where it slots in

Memory: survives restarts → reuses customer context.
Skills folder: versioned per customer in your repo.

Docs: hermes-agent.nousresearch.com/docs

Composio — 1000+ Tool Integrations, One SDK
TOOLBELT
Pre-built, authenticated tool layer for GitHub, Slack, Notion, Trello, Asana, Gmail, HubSpot, Stripe, etc. Handles OAuth, refresh tokens, scopes, and rate limits — the boring 80% of integration work.

Use case in the business

Instead of writing a Trello client + an Asana client + a Slack client, you ask Composio for tools by toolkit name and pass them straight to your agent. Per-customer auth is keyed by user_id — perfect for multi-tenant.

Install

bash 
pip install composio_openai
composio login
composio add trello asana gmail granola

Minimal working example

python 
from composio_openai import ComposioToolSet, App
from openai import OpenAI

toolset = ComposioToolSet()
tools = toolset.get_tools(apps=[App.TRELLO, App.GMAIL])

client = OpenAI()
resp = client.chat.completions.create(
    model="gpt-5.5",
    tools=tools,
    messages=[{"role": "user",
               "content": "Read today's emails and create Trello cards for any action items"}],
)
toolset.handle_tool_calls(resp, entity_id="customer_acme")

Where it slots in

Per-customer auth: pass entity_id="customer_xyz".
Tool discovery: agent can query toolkits at runtime.

Docs: docs.composio.dev · GitHub: ComposioHQ/composio

AgentMail — Email Identity Per Agent
EMAIL
API-first email built for two-way agent conversations. Each agent gets its own real inbox (its own @yourdomain address), with sockets for real-time inbound and SDKs for outbound + threading.

Use case in the business

Customer #7's SDR agent needs sdr@acme.yourdomain.com. Customer #8's support agent needs support@beta.yourdomain.com. AgentMail gives you thousands of inboxes programmatically — no Gmail account juggling.

Install

bash 
pip install agentmail
export AGENTMAIL_API_KEY=...

Minimal working example

python 
from agentmail import AgentMail

mail = AgentMail(api_key=os.environ["AGENTMAIL_API_KEY"])

# 1. provision an inbox for the new agent
inbox = mail.inboxes.create(username="sdr-acme", domain="yourdomain.com")

# 2. send
mail.messages.send(inbox_id=inbox.id, to="lead@prospect.com",
                   subject="Quick question", text="Hi — saw your post...")

# 3. listen for replies (websocket)
with mail.sockets.connect(inbox_id=inbox.id) as sock:
    for event in sock:
        if event.type == "message.received":
            agent.handle_reply(event.message)  # hand to Hermes

Where it slots in

Observability: every agent action leaves an email trail.
Failure routing: stuck agent emails meta-agent.

Docs: docs.agentmail.to · GitHub: agentmail-to/agentmail-python

2 · Tools the Agent Uses (Like a Human Would)

Granola — Meeting Notes
CAPTURE
AI notetaker that listens to Zoom/Meet/Teams and produces structured notes. No public REST API today — integrate via the desktop app's local file output, Zapier, or by reading the rendered note from Notion/Slack export.

Use case in the business

The agent reads the Granola note after every customer call, extracts action items, and pushes them to Trello/Asana — closing the loop between "we talked about X" and "X is in motion" without anyone touching a keyboard.

Practical pattern (Zapier bridge)

python 
# Granola → Zapier webhook → your endpoint → agent
from fastapi import FastAPI, Request
app = FastAPI()

@app.post("/granola/note")
async def note(req: Request):
    note = await req.json()  # {title, transcript, summary, action_items}
    for item in note["action_items"]:
        agent.tools.trello.create_card(
            list_id="todo",
            title=item["text"],
            description=f"From call: {note['title']}",
        )

If/when Granola ships an API, swap the webhook for a poll. The agent contract stays identical.

Trello / Asana — Where Work Actually Lives
TASKS
Task boards. Pick whichever the customer already uses — don't migrate them. Both have first-class Python SDKs and are wrapped by Composio.

Use case in the business

The agent is a board member. It creates cards from inbound email, moves cards through stages as work happens, comments with status, and assigns to humans when stuck. You should be able to look at the board and tell what the agent did this week.

Trello — direct SDK

python 
from trello import TrelloClient
tc = TrelloClient(api_key=KEY, token=TOKEN)
board = tc.get_board("acme-ops")
todo = next(l for l in board.list_lists() if l.name == "To Do")
todo.add_card(name="Follow up with prospect X", desc="From email thread #4421")

Asana — official client

python 
import asana
api = asana.ApiClient(asana.Configuration(access_token=PAT))
asana.TasksApi(api).create_task({"data": {
    "name": "Send proposal to Beta Inc",
    "projects": ["1234567890"],
    "notes": "Triggered by Granola call note 2026-05-13",
}}, opts={})

Reference: Asana/python-asana

Loom — Async Video Updates
VIDEO
Loom has no open REST API — it offers a Record SDK (browser/JS) and an oEmbed endpoint. The agent doesn't create Looms; you record one Loom per week per customer, and the agent embeds + distributes them.

Use case in the business

Each shipped workflow → 1 Loom = customer report + marketing asset. The agent tracks the Loom URL, embeds it in the weekly customer email, posts the public ones to your content channel.

Embed via oEmbed

python 
import requests

def loom_embed_html(loom_url: str) -> str:
    r = requests.get("https://www.loom.com/v1/oembed",
                     params={"url": loom_url})
    return r.json()["html"]  # <iframe ...>

# used inside the agent's weekly-report skill
report = f"""
<h2>Week of {date}</h2>
{loom_embed_html(loom_url)}
<ul>{action_items_html}</ul>
"""
agent.tools.email.send(to=customer.email, subject="Weekly update", html=report)

Docs: dev.loom.com

Superhuman — Email Surface (via MCP)
EMAIL UX
Superhuman doesn't expose a traditional REST API. Its MCP server exposes tools like send_draft, list_threads, and reply scheduling — usable directly from Claude Code or any MCP-aware agent. Business plan or higher.

Use case in the business

For agents handling your personal inbox (founder ops, replies to inbound, calendar coordination) Superhuman + MCP is faster and feels native. For per-customer agent inboxes, use AgentMail instead.

Wire it into Claude Code

bash 
claude mcp add superhuman -- npx -y @superhuman/mcp-server@latest
# then in a Claude Code session:
# > "draft replies to all unread emails from prospects, schedule for 9:03am tomorrow"

From Python (any MCP client)

python 
from mcp import ClientSession, StdioServerParameters

params = StdioServerParameters(command="npx",
                               args=["-y", "@superhuman/mcp-server@latest"])
async with ClientSession(params) as sh:
    await sh.call_tool("send_draft", {
        "to": ["customer@acme.com"],
        "subject": "This week's update",
        "html": report_html,
        "smart_send": True,
    })

Docs: Superhuman Mail MCP Server

Obsidian — Long-Term Agent Memory
MEMORY
Markdown vault. Each customer gets one. The agent reads/writes notes via the Local REST API plugin (or a thin MCP wrapper). Notes survive across sessions, are human-readable, and are diffable in git.

Use case in the business

The agent's "permanent memory" is a vault: customers/acme/people/jane.md, customers/acme/playbooks/sdr.md, customers/acme/decisions/2026-05-13.md. When a new agent boots in a new VM, it pulls the vault and is instantly up to speed.

Setup

bash 
# in Obsidian → Settings → Community plugins → install "Local REST API"
# enable it, copy the API key, note the port (default 27124 https)

Minimal working example

python 
import requests
BASE = "https://127.0.0.1:27124"
HDR  = {"Authorization": f"Bearer {OBSIDIAN_API_KEY}"}

# write a note (the agent's daily journal entry)
requests.put(f"{BASE}/vault/customers/acme/journal/2026-05-13.md",
             headers={**HDR, "Content-Type": "text/markdown"},
             data="## What I did today\n- Triaged 14 emails\n- Booked 2 demos\n",
             verify=False)

# search before acting (avoid duplicate work)
hits = requests.post(f"{BASE}/search/simple", headers=HDR,
                     params={"query": "Jane Smith"}, verify=False).json()

Plugin: coddingtonbear/obsidian-local-rest-api

3 · Builder Tools (Yours, Not the Agent's)

Codex — Build & Wire Integrations
CODE GEN
OpenAI's terminal coding agent (Rust, GPT-5.5). Good at small-to-medium scoped tasks: writing a new Composio toolkit shim, generating a webhook handler, drafting a Hermes skill. Cloud Codex runs the same tasks in parallel sandboxes for PR review.

Use case in the business

You'll write 1–3 new skills per customer per week. Codex is for those reps. Hand it the workflow doc and the existing skill folder; let it draft → you review → ship.

Install & use

bash 
npm i -g @openai/codex
# or: brew install --cask codex

cd ~/code/agents/acme
codex "add a skill that polls Granola webhook output in ./inbox and \
            creates Trello cards for action items"

Docs: developers.openai.com/codex/cli

Claude Code — Long-Horizon & Meta-Agent Logic
META-AGENT
Anthropic's CLI/SDK. Use Opus 4.7 for the meta-agent — the layer that takes a Telegram message and decides "spin up VM / patch a stuck worker / page the human." Long-horizon planning + tool use is its strength.

Use case in the business

The meta-agent is the only piece of code that sees all customers. It has access to Orgo, AgentMail, Telegram, and your skills repo. Codex/GPT-5.5 isn't the wrong model — Opus 4.7 is just less likely to fumble multi-step recovery flows.

Headless invocation (Python SDK)

python 
from anthropic import Anthropic
client = Anthropic()

resp = client.messages.create(
    model="claude-opus-4-7",
    max_tokens=4096,
    tools=[orgo_tool, agentmail_tool, telegram_tool],
    messages=[{"role": "user", "content": telegram_msg.text}],
)
# then dispatch tool calls — this is the meta-agent's main loop

Or as a subprocess (Claude Code CLI)

bash 
claude -p "VM for customer acme is unresponsive — restart it and email me when ready" \
       --allowedTools Bash,Read,Write \
       --output-format json

The meta-agent should run on your infra (a small server or Cloudflare Worker), not inside an Orgo VM — it's the supervisor.

MCPs — Live Context for Every Agent
CONTEXT
Model Context Protocol servers plug into any MCP-aware agent (Claude Code, Cursor, Codex). The video calls out three: Perplexity (live web answers), Context7 (up-to-date library docs), X (search/post on X).

Use case in the business

Without these, your agent hallucinates the API surface of every library you ask it to use. Context7 alone removes most "you used a deprecated method" bugs. Perplexity is the agent's web brain. X is for content distribution + competitive intel.

Add them to Claude Code

bash 
# Context7 — up-to-date docs for ~1000 libraries
claude mcp add context7 -- npx -y @upstash/context7-mcp@latest

# Perplexity — live web search
claude mcp add perplexity -- npx -y @perplexity/mcp@latest \
  --env PERPLEXITY_API_KEY=$PPLX_KEY

# X (Twitter) — search, post, DMs
claude mcp add x -- npx -y @x/mcp-server@latest \
  --env X_BEARER_TOKEN=$X_KEY

Use it from a prompt

text 
> use context7 to look up the latest agentmail-python sockets API,
  then write me a Hermes skill that listens for replies on inbox sdr-acme
  and routes anything containing "unsubscribe" to a Trello "Compliance" list.

Context7: upstash/context7 · MCP spec: modelcontextprotocol.io

How they actually wire together

You (Telegram) → Meta-agent (Claude Code + Opus 4.7, with Context7/Perplexity MCPs) → Orgo (spawn VM) → Hermes boots inside, loads its skills folder → Hermes uses Composio for app actions, AgentMail for its inbox, Obsidian for memory, and treats Trello/Asana/Granola/Loom/Superhuman as the human-visible surface. Failures bubble back via AgentMail → meta-agent → Telegram → you.

The Reference Architecture

Mental model

Telegram ⇢ Meta-Agent (Opus 4.7) ⇢ spawns Customer Workspace (Orgo) ⇢ which contains many Worker Agents (Hermes / GPT-5.5) ⇢ each with its own VM, email, tools, and an Obsidian memory vault.

Failures bubble up: a worker agent that gets stuck emails the meta-agent; the meta-agent either patches the VM in place or pings you on Telegram. Email + Telegram = the entire observability layer. No dashboards, no PagerDuty.

Your Implementation Guide (do this in order)

Pick one boring vertical you actually know

Marketing agencies, law, insurance, wholesale, real estate. Skip healthcare/finance for v1.

Write the offer in one sentence

"Unlimited AI employees for {industry} — $X/mo, all infra and changes included." Test it on 5 friendly buyers before you build anything.

Spin up Orgo and install Hermes on one VM

This is your "hello world" agent. Goal: it can read its inbox and reply.

Build the meta-agent in Claude Code (Opus 4.7)

Single job: take a Telegram message → spawn / install / restart worker agents on Orgo VMs.

Wire the customer-facing tools through Composio

Email (Agent Mail), Trello/Asana, Granola, Loom, Obsidian. Pick only the tools your first vertical actually lives in.

Choose models per job

GPT 5.5 for tool-using workers · GLM 5.1 / Kimmy for cheap parallel grunt work · Opus 4.7 for the meta-agent.

Onboard customer #1 with one workflow

One workspace, one agent, one job. Resist the urge to ship the whole "AI employee" on day one.

Run ops from Telegram, post a Loom every week

The Loom is both the customer report and your marketing. Each new workflow you ship becomes a piece of inbound content.

Scale by cloning workspaces, not by hiring

Each new customer = a duplicated Orgo workspace + a new row in your meta-agent's registry. One person, 20+ customers.

Watch-outs From the Live Demo

Reliability is the real product

Models will fail. VMs will hang. The reason customers pay $5K/mo is that you (via the meta-agent) make the failure invisible. Build the watchdog before you build the third workflow.

Don't oversell autonomy

Most customers only need 1–3 agents doing 1–3 workflows really well. "Unlimited" is a pricing posture, not a product roadmap.

Pick verticals where mistakes are cheap

Healthcare and finance punish errors with regulators. Marketing and wholesale punish errors with a polite email.
