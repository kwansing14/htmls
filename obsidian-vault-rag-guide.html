01

Why you can skip the vector DB

RAG — retrieval-augmented generation — is two moves: find the relevant text, then hand it to a model to answer from. A vector database only does the first move, and only one way of doing it.

A vector DB is an index optimised for approximate nearest-neighbour search over embeddings. That approximation earns its keep when you have millions of chunks and exact search is too slow. A personal Obsidian vault is hundreds, maybe a few thousand, notes. At that scale exact brute-force search is sub-millisecond, plain keyword search is instant, and very often the whole vault simply fits in the model's context window.

So "RAG without a vector database" isn't a hack — at vault scale it's the correct engineering call. The index is an implementation detail; retrieval is the only thing that matters, and there are four good ways to do it without standing up any infrastructure.

▎i

The whole pipeline: question → retrieve a few relevant notes → put them in the prompt → Claude answers, citing the notes. Everything below is variations on the retrieve step.

02

An Obsidian vault is already a great corpus

Most RAG tutorials spend half their length on chunking — slicing documents into retrieval-sized pieces without cutting mid-thought. An Obsidian vault hands you that for free: one note is one idea, written by a human, at the size a human thought it.

It also ships three signals most corpora lack — [[wikilinks]] that form a knowledge graph, #tags, and YAML frontmatter for metadata:

mddeploy.md

title: Deploying the worker

updated: 2026-06-18

# Deploying the worker

Push to main and CI runs `wrangler deploy`. See [[ci-cd]]
for the pipeline and [[secrets]] for the env vars.

`wrangler rollback` — details in [[incident-runbook]].

That one note is pre-chunked, tagged, dated, and linked to three neighbours. Retrieval can lean on every one of those signals — and the graph (tier 4) is the part a vector DB throws away.

03

Four tiers of retrieval

Ordered by how much machinery they need. Start at the top; only move down when answers aren't good enough. Most personal vaults never leave tier one or two.

tier 1 / context

Load it all

Vault fits in the context window? Send the whole thing and cache it. No retrieval, no index, no DB.

tier 2 / lexical

Keyword search

SQLite FTS5 or in-browser Pagefind. Retrieve the top matches by your own words. No embeddings at all.

tier 3 / semantic

Brute-force embeddings

One vector per note in a flat file, cosine similarity in NumPy. Semantic recall, still no vector DB.

tier 4 / graph

Graph & agentic

Follow [[links]] from a seed, or hand Claude search/read tools and let it walk the vault itself.

▎i

These compose. A strong production setup is often tier 2 to find seeds → tier 4 to expand along links → tier 1 caching on the answer call. No single tier is "the" answer.

04

Tier 1 — just load the whole vault

Claude's Opus and Sonnet models have a 1M-token context window. A typical personal vault is well under that, so the simplest "RAG" is no retrieval at all: concatenate every note, cache it once, and ask away. This is sometimes called cache-augmented generation.

First, measure — use the model's own tokenizer via count_tokens, never tiktoken (that's OpenAI's and undercounts Claude):

pyload_all.py
import anthropic, pathlib

client = anthropic.Anthropic()          # reads ANTHROPIC_API_KEY
vault  = pathlib.Path("~/Obsidian/vault").expanduser()

# one big corpus: every note, titled by filename
corpus = "\n\n".join(
    f"# {p.stem}\n{p.read_text()}" for p in vault.rglob("*.md")
)

# does it fit? (count with Claude's tokenizer, not tiktoken)
n = client.messages.count_tokens(
    model="claude-opus-4-8",
    messages=[{"role": "user", "content": corpus}],
).input_tokens
print(n, "tokens")              # under ~1M? skip retrieval entirely

If it fits, put the corpus in the system prompt with a cache breakpoint. The first call writes the cache; every later question reads it back at ~0.1× the price:

pyload_all.py
msg = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=1024,
    system=[{
        "type": "text",
        "text": corpus,
        "cache_control": {"type": "ephemeral"},   # cache the vault
    }],
    messages=[{"role": "user",
               "content": "What did I conclude about deploys?"}],
)
print(msg.content[0].text)
print(msg.usage.cache_read_input_tokens, "tokens served from cache")
▎i

Verify the cache is working. On the second identical-prefix request, usage.cache_read_input_tokens should be large. If it's zero, something volatile (a timestamp, a re-sorted note order) is changing the prefix bytes and busting the cache.

▎!

Minimum cacheable prefix on Opus 4.8 is 4096 tokens. A tiny vault won't cache (no error, just no savings) — and once you outgrow 1M tokens, stop loading everything and drop to tier 2.

05

Tier 2 — keyword retrieval, no embeddings

For your own notes, lexical search is shockingly good — you know your own vocabulary. Index the notes, retrieve the top matches, pass them to Claude. Two flavours depending on where the search runs.

Server-side · SQLite FTS5

FTS5 is a full-text engine built into SQLite — BM25 ranking, no extra dependency, the whole index in a single file:

pyfts_index.py
import sqlite3, pathlib

db = sqlite3.connect("vault.db")
db.execute("CREATE VIRTUAL TABLE IF NOT EXISTS notes "
           "USING fts5(path, title, body)")

for p in pathlib.Path("vault").rglob("*.md"):
    db.execute("INSERT INTO notes VALUES (?,?,?)",
               (str(p), p.stem, p.read_text()))
db.commit()

def search(q, k=8):
    return db.execute(
        "SELECT path, title, snippet(notes,2,'«','»','…',16) "
        "FROM notes WHERE notes MATCH ? ORDER BY rank LIMIT ?",
        (q, k),
    ).fetchall()

for path, title, snip in search("deploy worker rollback"):
    print(title, "—", snip)
▎i

Even lighter: skip the index and let ripgrep be your retriever — rg -l -i "deploy" vault/ | head -8. For a few hundred notes it returns instantly.

Static site · Pagefind in the browser

If your site is static HTML (like this page), Pagefind builds a chunked search index at deploy time and runs entirely client-side — no backend, no DB server. Render notes to HTML, then index:

⌘Terminal
# after rendering your vault to ./dist as HTML pages
npx pagefind --site dist
jssearch.js
const pagefind = await import("/pagefind/pagefind.js");
const { results } = await pagefind.search("how do I deploy the worker");

// hydrate the top hits → these become the notes you send to Claude
const top = await Promise.all(
  results.slice(0, 6).map(r => r.data())
);
// top[i].url · top[i].meta.title · top[i].excerpt
06

Tier 3 — embeddings without a vector DB

When keyword search misses synonyms — "shipping" vs "deploy" — you want semantic retrieval. You need embeddings for that, but you still don't need a vector database: at vault scale, brute-force cosine over a flat file is faster than the network round-trip to query one.

▎!

Claude has no embeddings endpoint. The Anthropic API is Messages, Batches, Files, and token counting — no /embeddings. For vectors, use a dedicated embedding model: Voyage AI (hosted, what Anthropic recommends) or a local model like nomic-embed-text via Ollama. Claude still writes the answer — it just doesn't make the vectors.

Embed every note once and save the matrix to a single .npz file — that flat file is your "database":

pybuild_index.py
import numpy as np, pathlib, voyageai

notes = [(p, p.read_text()) for p in pathlib.Path("vault").rglob("*.md")]

vo   = voyageai.Client()                    # reads VOYAGE_API_KEY
vecs = vo.embed([t for _, t in notes],
                model="voyage-3.5",
                input_type="document").embeddings

# local alternative — fully offline:
# import ollama
# vecs = ollama.embed(model="nomic-embed-text",
#                     input=[t for _, t in notes])["embeddings"]

np.savez("index.npz",
         M=np.array(vecs),
         paths=[str(p) for p, _ in notes])   # the whole "DB"

Query time: embed the question, take the dot product against the matrix, sort. No server, no ANN, just NumPy:

pysearch.py
import numpy as np, voyageai

idx = np.load("index.npz", allow_pickle=True)
M, paths = idx["M"], idx["paths"]
vo = voyageai.Client()

def search(query, k=6):
    q = np.array(vo.embed([query], model="voyage-3.5",
                          input_type="query").embeddings[0])
    sims = M @ q / (np.linalg.norm(M, axis=1) * np.linalg.norm(q))
    top  = sims.argsort()[::-1][:k]
    return [(paths[i], float(sims[i])) for i in top]
▎i

A few thousand notes is a matrix of a few megabytes; the dot product is microseconds. You'd only reach for a real vector DB (FAISS, pgvector, a hosted service) at hundreds of thousands of chunks.

07

Tier 4 — graph & agentic retrieval

This is the Obsidian-native tier — the one a vector DB can't do, because it throws the links away. Two approaches, and they stack on the tiers above.

Follow the links

Start from a handful of seed notes (from tier 2 or 3), then pull in everything they [[link]] to, one or two hops out. Often the linked note is exactly the missing context:

pyexpand.py
import re, pathlib

VAULT = pathlib.Path("vault")
LINK  = re.compile(r"\[\[([^\]|#]+)")      # [[note]] / [[note|alias]]

def expand(seeds, hops=1):
    seen, frontier = set(seeds), list(seeds)
    for _ in range(hops):
        nxt = []
        for path in frontier:
            for name in LINK.findall(pathlib.Path(path).read_text()):
                hit = VAULT / f"{name.strip()}.md"
                if hit.exists() and str(hit) not in seen:
                    seen.add(str(hit)); nxt.append(str(hit))
        frontier = nxt
    return seen      # seeds + linked neighbours → send all to Claude

Let Claude navigate

The fully agentic option: precompute nothing. Give Claude a search and a read tool and let it explore the vault the way you would — searching, opening a note, following a link, reading the next. The SDK's tool runner handles the loop:

pyagent.py
import anthropic, pathlib
from anthropic import beta_tool

client = anthropic.Anthropic()
VAULT  = pathlib.Path("vault")

@beta_tool
def search_notes(query: str) -> str:
    """Search the vault. Returns matching note titles, one per line."""
    hits = [p.stem for p in VAULT.rglob("*.md")
            if query.lower() in p.read_text().lower()]
    return "\n".join(hits[:10]) or "no matches"

@beta_tool
def read_note(title: str) -> str:
    """Read one note in full by its title."""
    p = VAULT / f"{title}.md"
    return p.read_text() if p.exists() else "not found"

runner = client.beta.messages.tool_runner(
    model="claude-opus-4-8",
    max_tokens=1024,
    tools=[search_notes, read_note],
    messages=[{"role": "user",
               "content": "Summarise what my notes say about deploys."}],
)

final = None
for message in runner:        # Claude searches & reads on its own
    final = message
print("".join(b.text for b in final.content if b.type == "text"))
▎i

This is the same shape as a coding agent's grep + read loop — the "index" is just the filesystem. Slower and pricier per question, but it needs zero precomputation and adapts to questions a fixed index would miss.

08

Wiring it into a website

For a static site, the clean split is: retrieve in the browser (Pagefind), generate on the server (a small serverless function that holds your API key). The browser never sees Claude directly.

step / 01

Browser searches

Pagefind runs client-side, returns the top note slugs + text for the question.

step / 02

POST to an edge fn

The page sends question + retrieved notes to your serverless endpoint.

step / 03

Server calls Claude

The function holds the API key, calls Claude, and streams the answer back.

▎!

Never ship your Anthropic API key to the browser. Retrieval can be client-side, but the model call must sit behind a serverless function or small backend. A key in front-end JS is a key anyone can copy from devtools.

tsapi/ask.ts — edge function
import Anthropic from "@anthropic-ai/sdk";

type Note = { title: string; text: string };

export default async function handler(
  req: Request,
  env: { ANTHROPIC_API_KEY: string },
) {
  const { question, notes } = await req.json();   // notes from Pagefind
  const client = new Anthropic({ apiKey: env.ANTHROPIC_API_KEY }); // stays server-side

  const stream = client.messages.stream({
    model: "claude-opus-4-8",
    max_tokens: 1024,
    system: [{ type: "text", text: SYSTEM,
               cache_control: { type: "ephemeral" } }],
    messages: [{
      role: "user",
      content: [
        ...notes.map((n: Note) => ({
          type: "document" as const,
          source: { type: "text" as const,
                    media_type: "text/plain" as const, data: n.text },
          title: n.title,
          citations: { enabled: true },           // answers cite notes
        })),
        { type: "text" as const, text: question },
      ],
    }],
  });

  return new Response(stream.toReadableStream()); // stream to the page
}
09

The grounded answer, with citations

The retrieve step found the notes; this is the G in RAG. The best-practice move for a notes site is to pass each retrieved note as a document block with citations enabled — Claude then points every claim back at the note it came from.

pyanswer.py
import anthropic
client = anthropic.Anthropic()

SYSTEM = ("Answer using only the provided notes. "
          "If they don't cover it, say so. Cite the notes you use.")

def answer(question, notes):            # notes: list[(title, text)]
    docs = [{
        "type": "document",
        "source": {"type": "text",
                   "media_type": "text/plain", "data": text},
        "title": title,
        "citations": {"enabled": True},
    } for title, text in notes]

    return client.messages.create(
        model="claude-opus-4-8",    # or claude-haiku-4-5 for cheap & fast
        max_tokens=1024,
        system=[{"type": "text", "text": SYSTEM,
                 "cache_control": {"type": "ephemeral"}}],
        messages=[{"role": "user",
                   "content": [*docs,
                               {"type": "text", "text": question}]}],
    )

msg = answer("How do I roll back a bad deploy?", retrieved)
for block in msg.content:
    if block.type == "text":
        print(block.text)
        for c in (block.citations or []):  # which note backed this span
            print("   ↳ from:", c.document_title)
▎i

Citations give you provenance for free — each answer span carries the note title (and character range) it came from, so you can render "source: [[incident-runbook]]" links right under the reply. Note: citations can't be combined with structured-output JSON mode.

On model choice: default to claude-opus-4-8 for the best reasoning over messy notes; switch to claude-haiku-4-5 when answers are simple and you want them cheap and fast. Cache the system prompt either way.

10

Keeping the index fresh

Tier 1 and tier 2 are cheap to rebuild — re-read the vault on each deploy, or on a file-watch. Tier 3 is the only one with real cost: re-embedding every note on every edit is wasteful. Fix it by hashing each note and re-embedding only what changed:

pyreindex.py
import hashlib, json, pathlib, numpy as np, voyageai

cache = json.loads(pathlib.Path("hashes.json").read_text() or "{}")
vo    = voyageai.Client()
stale = []

for p in pathlib.Path("vault").rglob("*.md"):
    text = p.read_text()
    h = hashlib.sha256(text.encode()).hexdigest()
    if cache.get(str(p)) != h:        # changed or new → re-embed
        stale.append((p, text)); cache[str(p)] = h

if stale:
    fresh = vo.embed([t for _, t in stale],
                   model="voyage-3.5", input_type="document").embeddings
    # …write fresh vectors into index.npz at their rows…
    pathlib.Path("hashes.json").write_text(json.dumps(cache))
    print(f"re-embedded {len(stale)} of {len(cache)} notes")
▎i

Hook it to a git pre-commit hook (if your vault is a repo) or a watchdog / fswatch watcher. A typical edit touches one or two notes, so re-indexing is near-instant.

11

Which tier do you actually need?

Three questions decide it: how big the vault is, whether keyword matching is enough, and whether you have a server. Map your answer here.

If…Use
vault fits in ~1M tokensTier 1 — load it all, cache it, skip retrieval
you search by your own wordsTier 2 — FTS5 (server) or Pagefind (static)
"find notes about X" semanticallyTier 3 — brute-force embeddings, flat file
answers should follow your linksTier 4 — graph hops, or let Claude navigate
static site, no backendPagefind in-browser + an edge fn for Claude
hundreds of thousands of chunksnow a vector DB earns its keep — different guide

The honest default for a personal vault: start at tier 1 or 2. The big context window and prompt caching make "retrieve a few notes and stuff them in" the right call far longer than you'd expect.

12

Where to go next

You have a working RAG pipeline with zero database infrastructure. From here, the upgrades that actually move quality:

  • Hybrid retrieval — run tier 2 and tier 3 together, merge the hits. Lexical catches exact terms, embeddings catch synonyms.
  • Add a reranker — fetch the top ~20, let a small reranker (Voyage's, or Claude itself) reorder to the best 6 before answering.
  • Expose the vault as an MCP server — give the search/read tools to any MCP-aware client, not just your own loop.
  • Prompt-cache the corpus — for a hot vault, pre-warm the cache so the first question of the day isn't the slow one.
  • Watch the answers, not the vectors — log questions Claude couldn't answer; those gaps tell you what to write next.
