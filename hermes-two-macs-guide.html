01

What Hermes Agent is

Hermes Agent is an open-source, self-improving AI agent from Nous Research that runs on personal devices, a VPS, or serverless infra — and talks to whatever model you point it at.

Unlike a plain chatbot, it has a learning loop: it writes its own skills from experience, refines them as it uses them, and keeps persistent memory across sessions (with full-text search over past conversations). The practical upshot: the more you use it on a recurring task, the faster and more reliable it gets.

It is model-agnostic. It can call cloud models (Anthropic, OpenAI, OpenRouter's 200+, Nous Portal, NVIDIA NIM, and more) — but it can equally point at a local model running on your own Mac through any OpenAI-compatible endpoint. That last part is the whole game here: your M1 and M2 become both the brain and the host.

▎i

The mental model. Hermes Agent = the orchestrator (tools, memory, skills, scheduling, messaging). A model server (llama.cpp / MLX / exo) = the brain. They speak over an OpenAI-compatible /v1/chat/completions URL — so the brain can live on the same Mac, the other Mac, or be spread across both.

03

Plan your two Macs

Before any commands: decide which Mac is the engine (holds the model, does the token-crunching) and which is the cockpit (runs the agent, tools, and messaging). Unified memory is the deciding factor, not the chip generation.

Reality-check your RAM

A model has to fit in unified memory alongside its context window. Here's what's realistic per machine:

Unified memoryWhat's realistic
8 GBBest as a cockpit / client only, or tiny 1–3B models. Use llama.cpp with a quantized KV cache.
16 GBRuns the recommended Qwen3.5-9B comfortably (~10–12 GB needed).
24–32 GB9B with a big context, or a 30B-class model.
M1 + M2 pooledA 70B-class model that fits on neither Mac alone — see §07.

Suggested roles

Put the heavier-RAM Mac on engine duty. If your M2 has more memory (common), it serves the model; the M1 stays light as the cockpit and messaging hub.

RoleMacRuns
EngineM2 (more RAM)The model server — llama-server / MLX / exo node, bound to the LAN.
CockpitM1Hermes Agent TUI + tools + hermes gateway; offloads thinking to the engine.

Three layouts follow, simplest to most ambitious: a LAN shared brain (§06), pooling both Macs with exo (§07), or two fully independent agents (§08).

▎i

Requirements (per the docs): Apple Silicon (M1 or later), 8 GB+ unified memory, and a model context window of at least 64K for Hermes compatibility. Both Macs on the same network; Thunderbolt between them is a nice-to-have for §07.

04

Install Hermes on both Macs

One line. The installer bundles its own Python 3.11, Node.js, ripgrep, and ffmpeg, so it won't fight your system setup. Run this on each MacBook.

▸terminal · run on M1 and M2
# installs hermes + its own python/node/ripgrep/ffmpeg
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash

# reload your shell so the `hermes` command is on PATH
source ~/.zshrc

# launch the setup wizard (model, tools, personality)
hermes setup

After setup, the everyday commands you'll reach for:

CommandWhat it does
hermesStart the interactive agent (the TUI).
hermes modelChoose your LLM provider / model — this is where you point at a local endpoint.
hermes toolsEnable or disable the built-in tools.
hermes gatewayStart the messaging gateway (Telegram, Discord, …).
hermes config setSet individual config values from the shell.
hermes doctorDiagnose a broken setup.
hermes updatePull the latest version.

Config lives in ~/.hermes/: API keys in ~/.hermes/.env, model/provider settings in ~/.hermes/config.yaml.

05

Stand up a local model

The engine Mac needs an OpenAI-compatible server. Two good options on Apple Silicon — pick by what you optimise for.

BackendBest atFirst tokenGeneration
llama.cppLow latency, tight RAM~67 ms~70 tok/s
MLX (omlx)Sustained long output~289 ms~96 tok/s

The recommended starter model is Qwen3.5-9B — strong reasoning, fits comfortably in 16 GB. We'll use llama.cpp for the concrete path because it's the most robust on memory-constrained Macs.

Step 1 — install the server
▸terminal · engine Mac (M2)
brew install llama.cpp
brew install huggingface-cli

# ~5.3 GB on disk, needs ~10 GB unified memory to run
huggingface-cli download unsloth/Qwen3.5-9B-GGUF \
  Qwen3.5-9B-Q4_K_M.gguf --local-dir ~/models
Step 2 — serve it on the network

The key flag is --host 0.0.0.0: it makes the server reachable from your other Mac, not just localhost. -c 131072 gives a 128K context (well above Hermes's 64K floor).

▸terminal · engine Mac (M2)
llama-server -m ~/models/Qwen3.5-9B-Q4_K_M.gguf \
  -ngl 99 # offload all layers to the Metal GPU \
  -c 131072 # 128K context window \
  -np 1 -fa on \
  --cache-type-k q4_0 --cache-type-v q4_0 # quantize KV cache (~75% less RAM) \
  --host 0.0.0.0 # <-- expose to the LAN
Step 3 — confirm it answers
▸terminal · either Mac
curl -s http://localhost:8080/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"Qwen3.5-9B-Q4_K_M.gguf",
       "messages":[{"role":"user","content":"Hello!"}],
       "max_tokens":50}' | jq .choices[0].message.content
▎i

Prefer the GUI / faster generation? Install the MLX app from omlx.ai ↗, download Qwen3.5-9B-mlx-lm-mxfp4, and start its server (defaults to http://127.0.0.1:8000). Same OpenAI-compatible API — just a different port and model name.

06

Topology A — one shared brain over the LAN

Recommended for most people. The M2 runs the model (§05). The M1 runs Hermes and simply points across the network at it. One agent, one memory, the heavy lifting on the bigger Mac.

Step 1 — find the engine's address

On the M2, get its LAN IP (or just use its .local name, e.g. m2-mbp.local):

▸terminal · engine Mac (M2)
ipconfig getifaddr en0 # e.g. 192.168.1.42  (Wi-Fi)
scutil --get LocalHostName # e.g. m2-mbp  → reachable as m2-mbp.local
Step 2 — point Hermes at it (on the M1)

Run hermes model, choose Custom endpoint, and give it the engine's URL and the model name:

▸terminal · cockpit Mac (M1)
hermes model
# → choose "Custom endpoint"
# Base URL:   http://m2-mbp.local:8080/v1   (or http://192.168.1.42:8080/v1)
# Model name: Qwen3.5-9B-Q4_K_M.gguf

hermes # start chatting — tokens are generated on the M2

Hermes automatically detects local and LAN endpoints and relaxes its streaming timeouts, so slower local generation won't trip a timeout. If you ever need to force it (very large contexts), set it explicitly:

▸~/.hermes/.env · cockpit Mac (M1)
# seconds; default 120, auto-bumped to 1800 for localhost/LAN
HERMES_STREAM_READ_TIMEOUT=1800
▎!

Trust the network you do this on. --host 0.0.0.0 exposes the model server to everyone on the LAN with no auth. Fine on your home network; on shared/public Wi-Fi, keep it on 127.0.0.1 and tunnel over SSH instead (ssh -L 8080:localhost:8080 you@m2-mbp.local).

07

Topology B — pool both Macs with exo

When you want a model bigger than either Mac can hold alone (a 70B-class model), exo ↗ shards it across the M1 and M2 and exposes a single OpenAI-compatible endpoint. Hermes then talks to the cluster as if it were one machine.

Step 1 — install exo on both Macs
▸terminal · run on M1 and M2
brew install --cask exo
# or grab the app from https://assets.exolabs.net/EXO-latest.dmg
Step 2 — start exo on each; they auto-discover

Launch exo on both Macs. They find each other on the network automatically — no config — and exo's topology-aware planner splits the model across whatever hardware it sees. For the lowest latency, connect the two Macs over Thunderbolt.

▸terminal · optional, both Macs (macOS 26.2+, Thunderbolt 5)
# enable RDMA over Thunderbolt 5 for the fastest device-to-device link
rdma_ctl enable
# then reboot both Macs (requires matching macOS versions)
Step 3 — point Hermes at the cluster

exo serves an OpenAI-compatible API at http://localhost:52415. Run hermes model on whichever Mac you're driving, pick Custom endpoint, and use the exo URL:

▸terminal · cockpit Mac
# sanity-check the cluster endpoint first
curl -N -X POST http://localhost:52415/v1/chat/completions \
  -H 'Content-Type: application/json' \
  -d '{"model":"mlx-community/Llama-3.2-1B-Instruct-4bit",
       "messages":[{"role":"user","content":"hi"}],"stream":true}'

hermes model
# → Custom endpoint → http://localhost:52415/v1
# → model name = the larger model you loaded in exo
▎i

How it works. exo uses MLX as its inference backend and MLX-distributed for communication, using each Mac's Metal GPU automatically. Expect ~6–10 tok/s on a 70B over Wi-Fi between two laptops — usable for chat and agent work, slower than a single-Mac 9B. Choose this layout for capability, not raw speed.

08

Topology C — two independent agents

Simplest conceptually: a complete Hermes on each Mac, each with its own local model, memory, and skills. Great when the two Macs do different jobs.

A productive split:

  • M2 = the always-on worker. Runs its own local model and hermes gateway, so you reach it from your phone (Telegram/Discord) and it executes scheduled automations unattended — digests, monitors, long jobs.
  • M1 = your interactive cockpit. Your daily-driver agent for chat, coding, and research in the TUI.

Each agent learns its own skills and keeps its own memory, so they specialise over time. They don't share a brain — but the gateway means you can hand a task to the M2 worker from anywhere, even from the M1.

▎i

Which topology? Want one capable assistant with the work on your bigger Mac → A. Want to run a model neither Mac fits alone → B. Want a dedicated always-on worker plus a separate daily driver → C. You can also start at A and add C's gateway later.

09

What you can actually do with it

Once a model is wired up, Hermes is a general-purpose agent. The high-value patterns:

use / chat

Reason, code, research

Drive it in the TUI for coding, analysis, and Q&A — with web browsing and code execution among its tools.

use / reach

Text it from your phone

hermes gateway connects Telegram, Discord, Slack, WhatsApp, Signal, or Email to the agent on your Mac.

use / cron

Unattended automations

Schedule a 7 a.m. news digest, a feed monitor, or a nightly cleanup — the worker Mac runs it while you sleep.

use / fork

Parallel workstreams

Spawn isolated subagents to research several threads at once, then collect the results.

use / skill

It builds its own skills

Repeat a task and it writes a reusable skill, improving on each run — your agent gets faster the more you use it.

use / make

Generate images & files

Image generation and file/code creation are built in, alongside 40+ other tools.

A few starter prompts

▸hermes › interactive
# scheduled, unattended (runs on the worker Mac)
every weekday at 7am, browse my 5 news sites and DM me a 10-bullet digest

# parallel research, then synthesis
spin up 3 subagents to compare Qwen, Llama, and Mistral for coding; summarise

# a repeatable job → becomes a saved skill
turn the CSVs in ~/reports into a weekly chart and email it to me

Handy in-session slash commands:

CommandDoes
/new · /resetStart a fresh conversation.
/modelSwitch model mid-session (e.g. local → cloud for a hard task).
/skills · /<name>Browse or run a learned skill.
/personalitySwitch the agent's persona.

▎i

Hybrid trick. Keep the local model as the default for everyday work (free, private), and add a cloud provider in ~/.hermes/.env. When a task is genuinely hard, /model over to a frontier model for that turn, then switch back.

10

Troubleshooting & gotchas

The handful of things that actually trip people up across two Macs.

SymptomFix
Can't reach engine from other MacConfirm the server was started with --host 0.0.0.0, both Macs are on the same network, and macOS firewall isn't blocking the port. Test with the curl from §05 using the LAN IP.
Context-window errors / truncationHermes needs 64K+ context. With llama.cpp use -c 131072. With Ollama, set OLLAMA_CONTEXT_LENGTH=64000 before starting the server — the API can't raise it afterward.
Streaming times out on slow generationSet HERMES_STREAM_READ_TIMEOUT=1800 in ~/.hermes/.env (auto-relaxed for LAN, but force it for huge contexts).
Out-of-memory / system swap thrashThe model + context doesn't fit. Use a smaller quant, shrink context, or move to exo pooling (§07). Re-check the RAM table in §03.
exo cluster won't formBoth nodes must be reachable on the network; for RDMA-over-Thunderbolt the macOS versions must match across devices. Plain networking works without it, just slower.
Something's just offRun hermes doctor — it diagnoses common setup problems — and hermes update to get the latest build.
