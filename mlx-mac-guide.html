01

Why MLX is fast on a Mac

MLX is an open-source array framework from Apple, designed from the ground up for Apple Silicon. If you've used NumPy or PyTorch, the API will feel familiar — the difference is what runs underneath.

The key trick is unified memory. On an M-series chip, the CPU and GPU share one pool of RAM. MLX arrays live in that shared memory, so operations can run on the CPU or GPU without ever copying data between them. That's why a MacBook can comfortably run models that would normally demand a discrete GPU.

  • Lazy & composable — computation is only run when you need the result, which lets MLX fuse and optimize operations.
  • Familiar API — mlx.core mirrors NumPy; mlx.nn mirrors PyTorch.
  • Multi-language — first-class Python, plus Swift, C++ and C bindings for native macOS / iOS apps.
  • Built for local agents — fast enough to run a tool-calling LLM loop in real time on your own hardware.
▎i

This guide companions Apple's WWDC26 session "Run local agentic AI on the Mac using MLX." We'll build up to exactly that — a working local agent — one command at a time.

02

Check your Mac is ready

MLX requires Apple Silicon (M1, M2, M3, M4 or later). It will not use the GPU on an Intel Mac. Confirm your chip and architecture first:

⌘Terminal
# What chip am I on?
sysctl -n machdep.cpu.brand_string
# → "Apple M3 Pro"  ✅

# Architecture must be arm64 (not x86_64)
uname -m
# → arm64  ✅

You'll also want a recent Python. macOS ships with one, but a clean install you control is better:

⌘Terminal
# Recommended: install Python via Homebrew
brew install python   # needs Homebrew → brew.sh
python3 --version     # Python 3.9+ (3.11+ ideal)
▎!

Use native arm64 Python. If you installed Python through Rosetta (x86), MLX won't engage the GPU. python3 -c "import platform; print(platform.machine())" must print arm64.

03

Install MLX

Always work inside a virtual environment so MLX and its model dependencies stay isolated from system Python.

⌘Terminal
# 1 — make a project + virtual environment
mkdir mlx-lab && cd mlx-lab
python3 -m venv .venv
source .venv/bin/activate

# 2 — install the core framework
pip install mlx

# 3 — install the LLM toolkit (this is what you'll use most)
pip install mlx-lm

Arrays, autograd, neural-net layers. The NumPy/PyTorch-style foundation.

pkg / mlx-vlm

04

Hello, MLX — a 10-second sanity check

Before touching any models, confirm MLX is alive and using the GPU. Create hello.py:

pyhello.py
import mlx.core as mx

# arrays live in unified memory — no device juggling
a = mx.array([1, 2, 3])
b = mx.array([4, 5, 6])

print("sum  :", a + b)
print("dot  :", mx.sum(a * b))
print("gpu  :", mx.default_device())   # → Device(gpu, 0)
⌘Terminal
python hello.py
# sum  : array([5, 7, 9], dtype=int32)
# dot  : array(32, dtype=int32)
# gpu  : Device(gpu, 0)   ✅ you're running on the GPU
05

Run a real LLM from the terminal

No code required. The mlx-lm package ships command-line tools that download a model from Hugging Face and run it locally. The mlx-community org hosts hundreds of ready-to-run, pre-quantized models.

One-shot generation

⌘Terminal
mlx_lm.generate \
  --model mlx-community/Llama-3.2-3B-Instruct-4bit \
  --prompt "Explain unified memory in one sentence." \
  --max-tokens 200

The first run downloads the model (a few GB) into your Hugging Face cache; subsequent runs are instant and fully offline.

Interactive chat

⌘Terminal
mlx_lm.chat --model mlx-community/Llama-3.2-3B-Instruct-4bit
# type messages, get replies, conversation history is kept
# ctrl-C / "exit" to quit
▎i

The -4bit suffix means the model is quantized to 4-bit weights — roughly a quarter of the memory, with minimal quality loss. Start small (3B), then scale up once you know it works.

06

The Python API

Two functions cover almost everything: load() and generate().

pychat.py
from mlx_lm import load, generate

model, tokenizer = load("mlx-community/Llama-3.2-3B-Instruct-4bit")

# apply the model's chat template for proper formatting
messages = [{"role": "user", "content": "Write a haiku about Apple Silicon."}]
prompt = tokenizer.apply_chat_template(
    messages, add_generation_prompt=True
)

text = generate(
    model, tokenizer,
    prompt=prompt,
    max_tokens=256,
    verbose=True,   # streams tokens + prints speed
)

Streaming tokens yourself

pystream.py
from mlx_lm import load, stream_generate

model, tokenizer = load("mlx-community/Llama-3.2-3B-Instruct-4bit")
prompt = tokenizer.apply_chat_template(
    [{"role": "user", "content": "Count to five."}],
    add_generation_prompt=True,
)

for chunk in stream_generate(model, tokenizer, prompt, max_tokens=128):
    print(chunk.text, end="", flush=True)
07

Serve it like the OpenAI API

This is the bridge to agentic apps. mlx-lm can expose your local model behind an OpenAI-compatible HTTP server — so any tool or SDK that speaks the OpenAI protocol can point at your Mac instead of the cloud.

⌘Terminal
mlx_lm.server --model mlx-community/Llama-3.2-3B-Instruct-4bit
# serving on http://localhost:8080

Now talk to it with the standard openai client — note the local base_url and a throwaway key:

pyclient.py
from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8080/v1",
    api_key="not-needed",        # local — no auth
)

resp = client.chat.completions.create(
    model="local",
    messages=[{"role": "user", "content": "Hello from my Mac!"}],
)
print(resp.choices[0].message.content)
▎i

Because it's OpenAI-compatible, the same server drops into tools like LangChain, LlamaIndex, or your own app — just swap the base URL. Your data never leaves the machine.

08

Build a local agentic loop

An "agent" is just an LLM that can call your functions (tools), read the results, and decide what to do next. With the local server running, here's a complete tool-calling loop — the heart of the WWDC26 demo, on your hardware.

pyagent.py
import json
from openai import OpenAI

client = OpenAI(base_url="http://localhost:8080/v1", api_key="x")

# 1 — a real Python function the model is allowed to call
def get_battery():
    import subprocess
    out = subprocess.run(["pmset", "-g", "batt"], capture_output=True, text=True)
    return out.stdout.strip()

# 2 — describe the tool to the model (JSON schema)
tools = [{
    "type": "function",
    "function": {
        "name": "get_battery",
        "description": "Get the Mac's current battery status.",
        "parameters": {"type": "object", "properties": {}},
    },
}]

messages = [{"role": "user", "content": "What's my battery at?"}]

# 3 — the agent loop: model thinks, maybe calls a tool, we feed results back
while True:
    r = client.chat.completions.create(
        model="local", messages=messages, tools=tools,
    )
    msg = r.choices[0].message
    messages.append(msg)

    if not msg.tool_calls:
        print(msg.content)        # final answer → done
        break

    for call in msg.tool_calls:
        result = get_battery()       # run the requested tool
        messages.append({
            "role": "tool",
            "tool_call_id": call.id,
            "content": result,
        })
▎!

Use a tool-capable model. Function calling needs a model trained for it — Llama 3.x, Qwen 2.5/3, and Mistral instruct variants all work. A base/completion model will ignore the tools field.

That's the whole pattern: prompt → model decides → tool runs → result returns → repeat until done. Add file access, web search, or shell commands as more tools and you have a genuine local agent — private, free to run, and offline.

09

Fine-tune with LoRA

Teach a model your own style or domain without retraining the whole thing. MLX has built-in LoRA / QLoRA fine-tuning that runs on the same Mac.

⌘Terminal
# data/ holds train.jsonl + valid.jsonl (chat-formatted examples)
mlx_lm.lora \
  --model mlx-community/Llama-3.2-3B-Instruct-4bit \
  --train \
  --data ./data \
  --iters 600

# then generate using your trained adapters
mlx_lm.generate \
  --model mlx-community/Llama-3.2-3B-Instruct-4bit \
  --adapter-path ./adapters \
  --prompt "..."

Pick a model for your RAM

Because of unified memory, your total RAM is roughly your VRAM budget. Leave headroom for macOS and apps. As a rule of thumb for 4-bit quantized models:

Your RAMComfortable size (4-bit)Good starting model
8 GBup to ~3BLlama-3.2-3B-Instruct-4bit
16 GB7B – 8BQwen2.5-7B-Instruct-4bit
24–32 GB13B – 14BQwen2.5-14B-Instruct-4bit
48–64 GB30B – 34BMixtral / Qwen-32B-4bit
96 GB+70B+Llama-3.3-70B-Instruct-4bit

Convert & quantize any Hugging Face model

Want a model that isn't pre-converted? Quantize it yourself:

⌘Terminal
mlx_lm.convert \
  --hf-path meta-llama/Llama-3.2-3B-Instruct \
  -q --q-bits 4 \
  --mlx-path ./my-llama-4bit

Troubleshooting

err / arch

GPU not used

You're on x86 Python under Rosetta. Reinstall native arm64 Python and recreate the venv.

err / oom

Out of memory

Model too big for your RAM. Drop to a smaller size or a lower bit-width (4-bit, even 3-bit).

err / slow

First run is slow

That's the model download. Later runs use the cached weights and run fully offline.

err / 404

Model not found

Check the exact repo id on Hugging Face. Gated models need huggingface-cli login.

err / cmd

mlx_lm not found

The venv isn't active. Run source .venv/bin/activate again in this shell.

err / tools

Tools ignored

Your model can't do function calling. Switch to Llama 3.x, Qwen 2.5/3, or Mistral instruct.
