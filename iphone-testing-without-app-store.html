01

Two ways onto a real phone

You don't need the App Store — or a paid developer account — to run the game on a physical iPhone. Pick the path that matches what you're testing.

Both paths talk to SpacetimeDB Maincloud over wss:// — that's the cloud Vite mode. Nothing connects to a local server, so you never have to expose or tunnel a backend to your phone. The only difference is whether the game runs inside the real native shell or just in mobile Safari.

opt / a · xcode

Sideload the app

Installs the native Capacitor shell (com.coorpg.game) onto your phone via Xcode. Free Apple ID; the install expires after 7 days.

opt / b · safari

Safari over LAN

Skip the native shell entirely. Open the dev server in mobile Safari on the same Wi-Fi. Instant hot-reload, zero install.

▎i

The game is a pure canvas web app, so Option B is the fastest loop for day-to-day work. Reach for Option A when you need to verify the real bundled WKWebView — touch input, safe areas, and performance can differ from Safari.

02

Option A — build the bundle & open Xcode

This installs the real native shell (ios/App, appId com.coorpg.game) onto your phone. No paid developer account needed — a free Apple ID works.

First, build the web bundle for cloud mode and sync it into the iOS project, then open the workspace in Xcode:

⌘Terminal
# build for Maincloud + copy the web build into ios/App
pnpm run ios:sync   # vite build --mode cloud + cap sync ios

# open the native project in Xcode.app
pnpm run ios:open   # opens ios/App in Xcode

Now connect the iPhone via USB. The first time, tap Trust on the phone when prompted, then unlock it.

▎!

Never build with xcodebuild from a headless terminal — it hangs. Always build and run from Xcode.app itself. The CLI build path is not supported here.

▎i

The sideloaded cert is good for 7 days. When it expires the app just won't launch — re-run it from Xcode to refresh another 7 days. No reinstall of the project needed.

03

Enable Developer Mode & set up signing

A physical device needs two one-time setups: Developer Mode on the phone, and a signing team in Xcode. Do these once per device / per machine.

On the iPhone — Developer Mode

This toggle only appears after Xcode has seen the device over USB at least once. Enabling it requires a reboot.

▎i

Settings → Privacy & Security → Developer Mode → on. The phone restarts; confirm the prompt after it boots back up.

In Xcode — automatic signing

Select the App target → Signing & Capabilities → check Automatically manage signing → pick your Personal Team. If no team is listed, add your Apple ID first:

xcode / accounts

Add your Apple ID

Xcode → Settings → Accounts → + → Apple ID. Your free account then shows up as a Personal Team.

xcode / signing

Pick the team

App target → Signing & Capabilities → Automatically manage signing → select your Personal Team.

ios / developer mode

Developer Mode

Settings → Privacy & Security → Developer Mode → on, then reboot the phone to apply.

04

Run it & trust the developer cert

With signing set, select your iPhone in Xcode's device dropdown and hit ⌘R. Xcode builds, installs, and launches the app on the phone.

On the first launch only, iOS blocks the app until you trust the certificate tied to your Apple ID:

▎i

Settings → General → VPN & Device Management → trust your Apple ID. After that, the app launches normally from the home screen — untethered — until the 7-day cert expires.

Re-deploy over Wi-Fi

You don't have to stay plugged in. In Xcode's Window → Devices and Simulators, enable Connect via network for your iPhone. After that, ⌘R deploys over Wi-Fi with no cable attached.

▎!

App won't open after a week? That's the expected 7-day expiry, not a bug. Plug in (or use network deploy), hit ⌘R from Xcode, and it's refreshed.

05

Option B — Safari over your LAN

The game is a pure canvas web app, so you can skip the native shell entirely while iterating. This is the fastest possible loop — changes hot-reload on the phone with no rebuild or reinstall.

Start the dev server in cloud mode, exposed to the local network with --host:

⌘Terminal
# dev server, Maincloud mode, listening on the LAN
pnpm run dev:cloud -- --host

# Vite prints a "Network:" URL on startup, e.g.
#   ➜  Network: http://192.168.1.42:5173/

On the iPhone — connected to the same Wi-Fi — open Safari to that Network URL: http://<your-mac-lan-ip>:5173. If Vite didn't print it, find your IP under System Settings → Wi-Fi → Details.

Optionally, tap Share → Add to Home Screen for a fullscreen, app-like icon that launches without Safari chrome.

▎i

Edits hot-reload instantly on the phone — no rebuild, no reinstall, no cable. This is the right loop for day-to-day gameplay tuning.

06

Which one to use

Use the lightweight loop for everyday work, and the native shell when you need to trust what you're seeing on the real device.

What you're doingUseWhy
Day-to-day gameplayOption BInstant hot-reload — no rebuild or reinstall between changes.
Touch, safe areas, perfOption ABehaves differently inside the bundled WKWebView than in Safari.
Verifying the real appOption AIt's the actual native shell users would install (com.coorpg.game).
Quick share / demoOption BAdd to Home Screen gives an app-like icon with zero signing.
▎i

A good rhythm: live in Option B all day, then do a pass in Option A before you trust a build — to catch input, safe-area, and performance differences the shell introduces.
