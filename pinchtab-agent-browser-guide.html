01

What PinchTab actually is

PinchTab is an open-source, standalone HTTP server that gives AI agents direct control over Chrome. One small Go binary turns a real browser into something an agent can navigate, read, and act on — over a CLI, a plain HTTP API, or MCP.

Most "let an LLM use a browser" setups are heavy: a framework, a headless driver, and screenshots piped back as images that burn through tokens. PinchTab takes the lean path. It describes each page as an accessibility tree with stable element references — e0, e1, e2 — instead of pixel coordinates, and lets the agent act on those refs.

  • Token-efficient — a page is ~800 tokens of structured text, not a screenshot (the docs cite roughly 5–13× cheaper).
  • Accessibility-first — agents click e5, not brittle x/y coordinates that break on every layout shift.
  • Self-contained — a single ~15 MB binary. No Node, no Python, no Selenium grid.
  • Local-first — binds to 127.0.0.1 by default and won't roam the public web until you allow it.
▎i

PinchTab is built for one operator on a managed machine — your laptop, a Raspberry Pi, a container. It is explicitly not a turnkey, internet-facing browser service. We come back to that in §09.

02

Five concepts, then everything reads the same

Learn these five nouns once and the CLI, the HTTP API, and the MCP tools all line up the same way.

core / server

Server

The control plane: profiles, instances, routing, dashboard. One per machine. pinchtab server.

core / instance

Instance

A running Chrome process. Exactly one profile per instance. Start it headless or headed.

core / profile

Profile

Persistent browser state — cookies, history, local storage. Reuse it to stay logged in.

core / tab

Tab

A single page inside an instance. Snapshots and actions always target a tab.

core / bridge

Bridge

A lean single-instance runtime that sits behind managed instances. pinchtab bridge.

▎i

For a quick one-off on your own machine you barely touch these — the CLI picks sensible defaults. They start to matter when you run several browsers in parallel or go remote.

03

Install it

Pick one route. Every method drops the same single binary onto your PATH.

⌘Terminal
# macOS / Linux — install script
curl -fsSL https://pinchtab.com/install.sh | bash

# Homebrew
brew install pinchtab/tap/pinchtab

# npm (global)
npm install -g pinchtab

Prefer a container? The image binds to loopback and keeps state in a volume:

⌘Docker
docker run -d \
  --name pinchtab \
  -p 127.0.0.1:9867:9867 \
  -v pinchtab-data:/data \
  --shm-size=2g \
  pinchtab/pinchtab

Confirm it landed:

⌘Terminal
pinchtab --help     # prints the command list → it's on your PATH
▎!

Piping a script into bash? If that makes you nervous (it should make everyone a little nervous), read install.sh first, or just use Homebrew / npm instead — same binary, package-manager hygiene.

04

Start the server

Three ways to run it. For everyday use, install it as a background daemon once and forget it's there.

CommandWhat it doesWhen
daemon installRegister the managed background serviceOne-time setup
daemonRun the background daemon (survives your shell)Daily use
serverRun server + dashboard in the foreground, logs liveFirst run / debugging
bridgeSingle-instance runtimeEmbedding / managed
⌘Terminal
pinchtab daemon install   # one-time: register the service
pinchtab daemon           # start it (backgrounds, persists)

# …or watch the logs live in your terminal instead:
pinchtab server           # foreground, ctrl-C to stop

Either way it now listens on http://127.0.0.1:9867 — that single port serves both the HTTP API and the dashboard.

▎i

Server vs bridge. server is the full control plane (many instances + dashboard); bridge is a single-instance runtime for when something else is doing the orchestrating. Most people just run the daemon, which manages a server for you.

05

Drive a page from the CLI

The fastest way to feel how PinchTab works — and it's the exact loop your agent will run later. It never changes: navigate → snapshot → act.

1 · Navigate

⌘Terminal
# open a tab, navigate, print a snapshot when it loads
pinchtab nav https://example.com --snap

# shortcut: navigate + snapshot in one verb
pinchtab quick https://example.com

2 · Snapshot — see the page the way the agent does

⌘Terminal
pinchtab snap -i -c   # interactive elements only, copied to clipboard
# →
# e0  link     "More information..."
# e1  button   "Accept"
# e2  textbox  "Search"

Every element gets a stable ref — e0, e1, e2. Those refs are the handles you act on, and they survive re-renders far better than CSS selectors or screen coordinates.

3 · Act

⌘Terminal
pinchtab click e1                 # click a ref
pinchtab fill e2 "pinchtab"       # set an input's value directly
pinchtab type e2 "pinchtab"       # type it key-by-key (fires key events)
pinchtab press Enter             # send a key
pinchtab text                    # extract the page's readable text

fill vs type: fill sets the value in one shot (fast, ideal for forms); type emits real keystroke events — reach for it when a field reacts to each character (autocomplete, validation as-you-type).

▎i

Add --json to any command for machine-readable output, and --tab <id> to target a specific tab when an instance has more than one. pinchtab tab lists them; pinchtab tab new <url> opens another.

06

Connect your AI agent (MCP)

This is the payoff. PinchTab speaks the Model Context Protocol, so any MCP-aware agent — Claude, Cursor, VS Code + Copilot, or your own — gets the whole browser as a set of callable tools.

Start the stdio MCP server (after the daemon/server is already up):

⌘Terminal
pinchtab mcp   # speaks MCP over stdio

Then register it with your client. For Claude Desktop, add this to its MCP config:

{ }claude_desktop_config.json
{
  "mcpServers": {
    "pinchtab": {
      "command": "pinchtab",
      "args": ["mcp"]
    }
  }
}

Cursor uses the same shape; VS Code + Copilot adds "type": "stdio". To point an agent at a remote PinchTab, pass the address and authenticate with a token:

⌘Terminal
export PINCHTAB_TOKEN=…                       # set on the client
pinchtab --server http://remote-host:9867 mcp

Once connected, the agent sees roughly 34 tools (all prefixed pinchtab_), grouped like so:

GroupTools (sample)
navigationnavigate · snapshot · screenshot · get_text
interactionclick · type · press · hover · focus · select · scroll · fill
keyboardkeyboard_type · insert_text · keydown · keyup
contenteval · pdf · find
tabslist_tabs · close_tab · health · cookies · connect_profile
waitwait · wait_for_selector · wait_for_text · wait_for_url · wait_for_load
networknetwork · network_detail · network_clear
dialogdialog
▎i

The agent runs the same loop you just did by hand: call pinchtab_snapshot to learn the refs, then pinchtab_click / pinchtab_fill with a ref or selector. Navigate → snapshot → act, on autopilot.

07

The HTTP API, by hand

Prefer to script it yourself? Everything is plain HTTP + JSON on localhost:9867. Here is the whole loop with curl and jq — profile → instance → tab → snapshot → action.

⌘drive.sh
# 1 — create a profile (persistent cookies / storage)
PROF=$(curl -s -X POST http://localhost:9867/profiles \
  -H "Content-Type: application/json" \
  -d '{"name":"work"}' | jq -r '.id')

# 2 — start a Chrome instance on that profile
INST=$(curl -s -X POST http://localhost:9867/instances/start \
  -H "Content-Type: application/json" \
  -d "{\"profileId\":\"$PROF\",\"mode\":\"headless\"}" | jq -r '.id')

# 3 — open a tab
TAB=$(curl -s -X POST http://localhost:9867/instances/$INST/tabs/open \
  -H "Content-Type: application/json" \
  -d '{"url":"https://example.com"}' | jq -r '.tabId')

# 4 — snapshot the interactive elements
curl "http://localhost:9867/tabs/$TAB/snapshot?filter=interactive"

# 5 — act on a ref
curl -X POST "http://localhost:9867/tabs/$TAB/action" \
  -H "Content-Type: application/json" \
  -d '{"kind":"click","ref":"e5"}'

The CLI is just a thin client over exactly these endpoints — anything you can do with pinchtab … you can do with a curl.

08

Headless, headed & staying logged in

Two modes, and one good reason to care about profiles.

  • Headless ("mode":"headless") — no visible window. Fast and cheap; perfect for servers, CI, and a Raspberry Pi.
  • Headed ("mode":"headed") — a real Chrome window you can watch. Great for debugging an agent, or doing a first interactive login.

Here's the trick for authenticated automation: log in once in a headed instance. Because the profile persists cookies and storage, every later instance on that profile — even headless ones — is already signed in. No credentials ever go into your agent's prompts.

⌘Terminal
# headed, so you can log in by hand the first time
curl -s -X POST http://localhost:9867/instances/start \
  -H "Content-Type: application/json" \
  -d '{"profileId":"work","mode":"headed"}'

# …log in once in the window, then close it.
# later runs reuse profile "work" — already authenticated, headless.
▎i

On ARM64 (Apple Silicon, Raspberry Pi) PinchTab detects a native Chromium automatically — the same binary and the same commands just work.

09

Security: local-first by design

PinchTab hands real browser control to anything that can reach it. Treat the API, dashboard, and MCP server as privileged operator surfaces — not public endpoints. The good news: the defaults are locked down.

  • Loopback only — binds 127.0.0.1, so nothing off-box can reach it out of the gate.
  • IDPI on — by default the agent may only browse locally hosted sites; it can't wander the public web until you explicitly allow hosts.
  • Sensitive endpoints off — privileged endpoint families are disabled by default.
  • Attach off — hooking into arbitrary existing Chrome processes is disabled by default.
  • Cookies — the Secure flag is applied on HTTPS.
▎!

Do not expose it to the internet. The dashboard, HTTP API, and MCP server are full control of a browser running as you. PinchTab is explicitly not a turnkey internet-facing service. If you must go remote, put it behind a VPN or tunnel and require PINCHTAB_TOKEN.

Inspect and adjust the posture interactively:

⌘Terminal
pinchtab security          # interactive security overview
pinchtab config get        # view current configuration
pinchtab config set …      # change a setting (e.g. allow a host)

Troubleshooting

net / refused

Connection refused

Nothing is listening on :9867. The server/daemon isn't running — start pinchtab daemon (or server).

web / blocked

Public URL won't open

That's IDPI doing its job. Allow the host in config if you really mean to leave localhost.

ref / stale

A ref stopped working

Refs like e5 are per-snapshot. The page re-rendered — take a fresh snap and use the new refs.

auth / 401

Remote calls rejected

Set PINCHTAB_TOKEN on the client so it matches the remote server's token.

cmd / path

command not found

The binary isn't on your PATH. Re-check the install step and restart your shell.

arm / chromium

No browser on a Pi

PinchTab auto-detects Chromium on ARM64 — install Chromium via your package manager if it's missing.
