Project Deep Dive

Xikipedia: Wikipedia, restyled as a doom-scrollable feed

A single-page web app by rebane2001 that turns Simple Wikipedia into a TikTok-style feed — and uses a pure-JavaScript, ~50-line recommender to learn what you like, on-device, with zero machine learning.

📦 Repo: github.com/rebane2001/xikipedia 🌐 Live: xikipedia.org 📜 License: AGPLv3 🧠 Algorithm: client-side, no ML
01 · What it is

A pseudo social-media feed wrapped around Wikipedia

Xikipedia is a single HTML file (plus a small service worker and a compressed JSON blob) that loads ~all of Simple Wikipedia into the browser and presents each article as a "post" in an infinite vertical feed — title, short summary, optional image, a Like button.

The author calls it a demonstration: proof that even a basic, non-ML algorithm with no signals from other users can rapidly learn what you engage with and feed you more of it. There is no backend, no telemetry, no recommendation service. Everything runs on your device, in JavaScript, against a local dataset.

Once loaded, the whole app works offline and can be installed as a PWA. Profiles, algorithm state, and stats persist in browser storage.

02 · Data pipeline

From Wikipedia dump to a 1-file dataset

process_data.py in the repo ingests two MediaWiki dumps and produces a single smoldata.json.br (Brotli-compressed JSON) that the page slurps in at startup.

XML articles dump
simplewiki-*-pages-articles-multistream.xml.bz2
→
process_data.py
parse wikitext, strip refs, extract intro, categories, thumbnail
→
smoldata.json.br
pages + categories + pagelinks, Brotli-compressed

For each article the script extracts: the first ~300 chars of clean text, the list of categories (Wikipedia's own category tree), a candidate thumbnail filename, and a "disambiguation" flag. A separate pass over the pagelinks SQL dump joins each article with its outgoing links. The result is a flat list of pages, each carrying its categories + pagelinks — those are the "tags" the algorithm will reason about.

Why both? Categories give broad topical signal (e.g. "Programming languages"); pagelinks give fine-grained associative signal (the article about Python links to Guido van Rossum, Monty Python, scripting language). Treating both as flat features lets the scoring system learn taste across topics and across vibes.
03 · The algorithm

A category-scoring system that fits on a napkin

Core idea

Every category and every linked page is a feature. The user has a single dictionary, categoryScores, mapping each feature → an integer score. When you engage with a post, every feature attached to that post gets the same delta added to its score. When the next post is selected, its score is the sum of the scores of all its features.

That's the whole model. No vectors, no embeddings, no gradients. Just "if you liked posts tagged X, posts tagged X are now worth more points."

Engagement signals

Each interaction adds the same delta to every category and pagelink that post carries:

ActionΔ scoreIntuition
Scroll past (no interaction)−5Mild dislike. Applied to every post the moment it shows.
Like+50 + 4 × posts_since_last_likeThe longer you went without liking, the more this Like counts. Punishes bait, rewards rare enthusiasm.
Tap article title/body+75Opens the Wikipedia article in a new tab. Strong intent signal.
Tap image+100Strongest signal. You stopped to look.
JSfunction engagePost(post, amount) { post.allCategories.forEach(e => categoryScores[e] = (categoryScores[e] ?? 0) + amount); return amount; }

Three lines. allCategories is the post's Wikipedia categories plus its outgoing pagelinks. Every one of them moves by the same amount. That's the entire credit-assignment mechanism.

Per-post base score

Before category scores are summed, each candidate gets a small base score:

  • +5 if the post has a thumbnail image. Discovery bias — posts with pictures feel better in a feed, so the system nudges them up slightly when nothing else is known.
  • (3seen − 1) × −50,000 penalty for repeats. Anti-repeat — first time: 0 penalty. Second time: −100,000. Third: −1,300,000. It explodes geometrically so re-runs essentially can't win unless every category is on fire.
  • −1,000 starting score for the "given names" and "surnames" categories. Prior — without this, almost every biography article would dominate (nearly all of them carry these two categories).
Note: the README says the repeat penalty is −5,000, but the shipped index.html uses −50,000. Same shape (geometric), one order of magnitude harsher than the docs claim.

Candidate generation: 10,000 random pulls

For each post in the feed, the algorithm does not score the whole dataset. Instead, it samples 10,000 random pages from the dataset and scores only those. This is a deliberate efficiency-vs-quality tradeoff: scoring all ~250k Simple Wikipedia pages on every scroll would be prohibitively expensive in the browser, while 10k is plenty for a stochastic feed.

JSconst potentialPosts = [...Array(10000)] .map(e => pagesArr[Math.floor(Math.random() * pagesArr.length)]) .map(post => { const initialScore = (post.thumb ? 5 : 0) + (3**(post.seen ?? 0) - 1) * -50000; post.score = [...post.allCategories] .reduce((sum, cat) => sum + (categoryScores[cat] ?? 0), initialScore); return post; });

Selection: three dice, three personalities

From the 10,000 candidates, the algorithm picks the next post using one of three strategies, chosen randomly:

Weighted random sampleProbability ∝ score
40%
Highest-scoring postPure greedy
42%
Completely randomUniform over the 10k
18%

The three branches are an explicit exploit / explore mix:

  • Weighted random (40%) — picks proportional to (score − minScore), so higher-scoring posts are more likely but everything has a shot. This is the "soft personalization" branch. Soft exploit
  • Greedy (42%) — picks the single highest-scoring candidate out of 10,000. This is what makes the feed feel like it's "locking in" on a topic. Hard exploit
  • Random (18%) — totally ignores scores. Keeps the feed from collapsing into a single subject and surfaces things you'd never search for. Explore
Implementation detail: in the actual code, the second and third branches are nested under else if (Math.random() > 0.3). That is, after the 40% weighted branch fails, the remaining 60% splits 70/30 → 42% greedy / 18% random. The README's numbers match exactly.
JSif (Math.random() < 0.4) { // weighted random: probability proportional to (score - minScore) const minScore = Math.min(...potentialPosts.map(e => e.score)); const maxScore = potentialPosts.reduce((s, p) => s + p.score - minScore, 0); const target = Math.random() * maxScore; let running = 0; while (running < target && potentialPosts.length) { bestPost = potentialPosts.pop(); running += bestPost.score - minScore; } } else if (Math.random() > 0.3) { // greedy: highest-scoring candidate wins potentialPosts.forEach(p => { if (p.score > highestScore) { bestPost = p; highestScore = p.score; } }); } // else: bestPost stays as the first random candidate (the 18% explore branch)
04 · Why it works

How a 50-line algorithm imitates "the algorithm"

⚡

Dense signal per click

Because every category and pagelink on the post gets the same delta, a single Like updates dozens of features at once. After 5 likes, the model already has hundreds of category votes — enough to clearly prefer "World War II battles" over "Pokémon Gen II."

🔁

Constant negative feedback

The default action — scrolling past — is also a signal (−5). This is the secret sauce: you don't need explicit dislikes when the act of moving on is itself data. Lots of "scrolled past biography" votes drive down the biography features fast.

🎲

Hard-coded exploration

18% pure-random + 40% weighted-random means the feed never goes purely greedy. Even with strong preferences, ~58% of posts come from non-deterministic branches, so accidental discovery is baked in.

🧱

Crushing repeat penalty

The geometric −50,000 × (3seen−1) is brutal by design. It guarantees that the greedy branch can't get stuck in a 2-post loop, which is the failure mode this kind of system is most prone to.

🚫

Prior corrections

"Given names" and "surnames" start at −1,000 because virtually every biography on Wikipedia carries them. Without this hand-tuned prior, the model would conclude after one Like that you love biographies of everyone.

📈

Like ramp

Liking is worth 50 + 4·posts_since_last_like. A Like after scrolling 30 posts is worth +170 — a strong, rare signal. Spamming Likes flattens to +50. The system trusts you more when you're picky.

05 · One iteration, end to end

What happens between two scrolls

You see post N and scroll past

engagePost(N, -5) runs. Every category and pagelink attached to post N has its score decremented by 5.

The page calls getNextPost()

10,000 random pages are sampled from pagesArr. For each one, the algorithm computes initialScore + Σ categoryScores[cat].

One die is rolled

With Math.random() < 0.4 the weighted-random branch fires. Otherwise a second die decides greedy (42%) or fully random (18%).

The chosen post is rendered

Its seen counter increments by 1 — guaranteeing a 100,000-point penalty if it ever shows up in a future candidate pool.

If you Like / tap / open the image…

engagePost(post, +50…+100) runs and dozens of features shift in your favor at once. The next call to getNextPost() will land somewhere related — but not too related, thanks to the 18% random branch.
06 · Tech stack

What's actually shipped

  • index.html — a single ~55 KB file containing all markup, all CSS, and all the JavaScript including the algorithm. No build step. HTML 87%
  • sw.js — a service worker that caches the app shell + dataset for full offline use and PWA installability.
  • app.webmanifest — PWA manifest so iOS/Android/desktop can install Xikipedia as a native-feeling app with its own icon.
  • smoldata.json.br — Brotli-compressed dataset of pages, categories, and pagelinks. Loaded once, decompressed in-browser.
  • process_data.py — the only Python in the project. Parses MediaWiki XML + pagelinks SQL into the JSON dataset. Python 8%
  • localStorage — holds profiles, categoryScores, seen posts, liked posts, time-spent stats. No remote storage.

The split on GitHub — HTML 87.2%, Python 7.8%, JS 5.0% — slightly overstates how little JS there is, because all the JS lives inside index.html and gets counted as HTML.

07 · Takeaways

What this project is really saying

Xikipedia is, on one level, a joke: Wikipedia as a brain-rot feed. But the source code is a precise, readable argument that "the algorithm" — the thing platforms talk about as if it were a vast inscrutable AI — can be reproduced in tens of lines of vanilla JS, with no neural net, no user clustering, and no data leaving the device.

The ingredients are mundane: tag your content, pick a few engagement signals, give every signal a hand-tuned weight, add a repeat penalty, and mix exploit with explore via a coin flip. Run it locally. The result still feels eerily personalized within a dozen scrolls.

It's a useful piece of evidence the next time you hear that recommender systems must be opaque, must be ML-driven, or must collect data on you to work.
