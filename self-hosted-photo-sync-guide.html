01

How the system fits together

Four pieces, each doing one job: your phone captures, a small server at home stores and indexes, a private VPN carries the photos when you're out, and a separate backup protects it all.

◈architecture
┌──────────────┐   home Wi-Fi or Tailscale   ┌─────────────────────────────┐
│    iPhone    │ ──────── auto-upload ─────▶ │   Raspberry Pi · Docker     │
│  Immich app  │                             │   immich-server     :2283   │
└──────────────┘                             │   machine-learning          │
                                             │   postgres · redis          │
                                             └──────────────┬──────────────┘
                                                            ▼
                                             ┌─────────────────────────────┐
                                             │  /library — your originals  │
                                             └──────────────┬──────────────┘
                                                nightly job ▼
                                             external disk + offsite copy
  • The Immich iOS app watches your camera roll and uploads new photos and videos automatically — full-resolution originals, including HEIC and Live Photos.
  • The server runs Immich in Docker: a web app that looks and feels like Google Photos, with a timeline, albums, face recognition, and search — all computed locally.
  • Tailscale gives your phone a private, encrypted path back to the server from any network, without opening a single port on your router.
  • A nightly backup job copies the library and database somewhere else, because a sync target is not a backup (more on that in section 09).
▎i

Sync here means one-way upload. Deleting a photo on your phone does not delete it from the server, and vice versa. That's a feature: the server becomes the permanent archive, and your phone becomes a device you can safely wipe.

03

The hardware: a Raspberry Pi

A Raspberry Pi makes a genuinely good Immich server: silent, ~5 W at the wall, and small enough to live on top of the router. Uploads and day-to-day browsing are easy work for it — the only thing it does slowly is machine-learning indexing, which just means the first face-recognition pass takes longer. The shopping list:

kit / board ★

Raspberry Pi 5, 8 GB

Get the 8 GB model — Postgres, Redis and the ML container all want breathing room. A Pi 4 (8 GB) works too, just slower everywhere; 4 GB boards swap themselves to death during ML jobs.

kit / storage

SSD — never an SD card

An NVMe drive on the official M.2 HAT+ is ideal; a USB 3 SSD is nearly as good. SD cards are too slow for Postgres and wear out under constant writes — boot from the SSD too if you can.

kit / cooling

Active Cooler + case

ML indexing pins all four cores for hours. The official Active Cooler (or any fan case) keeps the Pi 5 from thermal-throttling mid-job.

kit / power

Official 27 W USB-C PSU

A Pi 5 plus an NVMe drive is power-hungry by Pi standards, and an underpowered supply shows up as mysterious SSD dropouts. Don't improvise this part.

Sizing the storage

Plan for roughly 2× your phone's photo library, because Immich stores your originals plus generated thumbnails and transcoded video previews. A 256 GB iPhone that's mostly full is comfortable on a 1 TB SSD with years of headroom.

▎!

One disk is not enough for the whole system. Budget for a second disk (a cheap external USB drive is fine) to receive nightly backups. A single SSD holding the only copy of your photos is the situation you're trying to escape.

Plans change? Any used mini PC, old laptop, or Synology-style NAS runs this exact guide unmodified — and chews through the ML indexing 5–10× faster. Nothing that follows is Pi-specific except the flashing step.

04

Prep the Pi

Flash Raspberry Pi OS Lite (64-bit) onto your SSD (or SD card, if you must) with the official Raspberry Pi Imager. Before writing, choose Edit Settings in the imager: set the hostname to raspberrypi, enable SSH, and create your username and password. The Pi then boots headless straight onto your network — no monitor or keyboard needed. SSH in and set up Docker:

⌘ssh — raspberrypi
# keep the box current
sudo apt update && sudo apt upgrade -y

# install Docker Engine + Compose plugin (official convenience script)
curl -fsSL https://get.docker.com | sh

# let your user run docker without sudo (re-login afterwards)
sudo usermod -aG docker $USER

# sanity check
docker compose version   # → Docker Compose version v2.x  ✅
▎i

64-bit is non-negotiable. Immich's ARM images are arm64 only. Raspberry Pi OS Lite (64-bit) or Ubuntu Server for Pi both work; the legacy 32-bit OS does not. Prefer Ethernet over Wi-Fi for the Pi if the router is within reach — uploads will thank you.

Give the Pi a fixed address

Your phone will be configured with the Pi's address, so that address must not change. In your router's admin page, find the Pi's DHCP lease and mark it as reserved / static (e.g. 192.168.1.50). Two minutes now saves confusing "server unreachable" moments later.

Prepare the photo location

If the library will live on a separate USB SSD, mount it somewhere permanent — for example /mnt/photos — and add it to /etc/fstab so it survives reboots. If you're booting the Pi from one big NVMe/SSD, just create the folder: sudo mkdir -p /mnt/photos && sudo chown $USER /mnt/photos.

05

Install Immich

Immich ships an official Docker Compose file, and every image is published for ARM64 — the standard setup runs on the Pi unchanged. You download two files, edit three lines, and start it.

⌘ssh — raspberrypi
# 1 — a home for the stack
mkdir ~/immich-app && cd ~/immich-app

# 2 — official compose file + environment template
wget -O docker-compose.yml \
  https://github.com/immich-app/immich/releases/latest/download/docker-compose.yml
wget -O .env \
  https://github.com/immich-app/immich/releases/latest/download/example.env

Open .env and set where photos live, where the database lives, your timezone, and a real database password:

env.env
# where your original photos/videos will be stored — point this at your big disk
UPLOAD_LOCATION=/mnt/photos/immich-library

# where the Postgres data lives — keep this on an SSD
DB_DATA_LOCATION=./postgres

# your timezone, for correct timestamps in logs and jobs
TZ=Asia/Singapore

# pin a version instead of chasing "release" — see section 10
IMMICH_VERSION=release

# change this before first start — it can't be changed easily later
DB_PASSWORD=pick-a-long-random-password

Then bring the stack up:

⌘ssh — raspberrypi
docker compose up -d

# watch it come alive (first start pulls a few GB of images)
docker compose ps
# immich_server            healthy   0.0.0.0:2283->2283/tcp
# immich_machine_learning  healthy
# immich_postgres          healthy
# immich_redis             healthy
▎i

Four containers, one job each: immich_server is the web app and API, immich_machine_learning runs face recognition and smart search, postgres holds all metadata, and redis queues background jobs. Your actual photo files never live inside a container — they're plain files in UPLOAD_LOCATION.

06

First-run setup

From any browser on your home network, open:

🌐browser
http://192.168.1.50:2283   # ← your Pi's static IP
  1. Create the admin account. The first sign-up becomes the administrator. Use a strong password — this account can see everything.
  2. Skim the onboarding. The defaults are sensible; you can accept them all.
  3. Optional: add family. Under Administration → Users, create a separate account per person. Each phone syncs into its own library; you can still share albums between accounts.

Poke around the web app for a minute — upload a test photo by dragging it into the browser. If it appears in the timeline, the server side is done.

07

iPhone auto-sync — the main event

Install Immich from the App Store, then wire it up. On home Wi-Fi first; remote access comes next section.

  1. Connect: open the app and enter your server's address — http://192.168.1.50:2283 — then log in with the account you created.
  2. Choose what backs up: go to the cloud icon → Backup, and select the albums to sync. Recents covers the whole camera roll.
  3. Turn on automatic backup: enable foreground backup (uploads whenever the app is open) and background backup (iOS periodically wakes the app to upload).
  4. Let iOS help: in Settings → Apps → Immich, make sure Background App Refresh is on. Leave the app installed and logged in — that's it.
  5. Watch the first sync: the initial upload of a full camera roll takes hours — leave the phone plugged in, on Wi-Fi, with the app open. Everything after that is incremental and automatic.
▎!

Know what iOS allows. Apple doesn't let any app upload continuously in the background — Immich included. Background backup runs opportunistically (typically when the phone is idle, charging, or on Wi-Fi), so new photos may take a few hours to appear on their own. Opening the app for a few seconds always flushes the queue immediately. This is an iOS platform limit, not an Immich bug.

HEIC, Live Photos, and originals

Immich uploads the untouched original files — HEIC stays HEIC, Live Photos sync as their photo + video pair and play in the web app, RAW and ProRes come across intact. Thumbnails and browser-friendly previews are generated server-side, so nothing is lost in translation.

▎!

Using iCloud's "Optimise iPhone Storage"? Then many of your originals exist only in iCloud, and the phone holds low-res stand-ins. Immich must download each original from iCloud before uploading it, which makes the first sync slow and battery-hungry. Do the initial backup plugged in and on Wi-Fi, and give it time. Once Immich holds everything, you can decide whether to keep paying for iCloud at all.

08

Sync from anywhere with Tailscale

Right now your phone can only reach the server on home Wi-Fi — photos taken elsewhere just queue up until you get home. Tailscale removes that limit: a free, private WireGuard mesh between your devices, with no ports opened on your router and no domain to buy.

⌘ssh — raspberrypi
# install tailscale on the pi
curl -fsSL https://tailscale.com/install.sh | sh

# join your tailnet (prints a login URL — open it once)
sudo tailscale up

# note the pi's tailscale name / IP
tailscale status
# 100.101.102.103   raspberrypi   linux   -
  1. On the iPhone: install the Tailscale app, sign in to the same account, and toggle the VPN on. Leave it on — it's near-zero battery cost and only carries traffic to your own devices.
  2. Enable MagicDNS in the Tailscale admin console so your Pi has a stable name like raspberrypi.
  3. Update the Immich app's server URL to http://raspberrypi:2283 (or the 100.x.y.z IP). Now backup works from cellular, the office, or a hotel on the other side of the world, exactly as it does at home.
▎i

Is plain http safe here? Yes — inside a tailnet, every packet is already end-to-end encrypted by WireGuard before it leaves the phone. Nothing traverses the internet unencrypted, and nothing on your router is exposed.

Alternative: a public URL with Cloudflare Tunnel

If you want to share album links with people outside your tailnet, a Cloudflare Tunnel gives Immich a real HTTPS address on a domain you own (e.g. photos.example.com) — still without opening router ports. The trade-off: your login page becomes reachable by the whole internet, so use a strong password, keep Immich updated, and consider putting Cloudflare Access in front. For a personal photo archive, Tailscale-only is the safer default.

09

Back up the backup

Your server is now the only full copy of your photo history — which makes it a single point of failure. Two things must be protected: the library folder (the original files) and the Postgres database (albums, faces, edits, metadata).

shimmich-backup.sh
#!/bin/bash
# nightly immich backup → external disk mounted at /mnt/backup
set -euo pipefail
DEST=/mnt/backup/immich
mkdir -p "$DEST"

# 1 — dump the database (albums, people, metadata)
docker exec -t immich_postgres \
  pg_dumpall --clean --if-exists --username=postgres \
  | gzip > "$DEST/immich-db-$(date +%F).sql.gz"

# 2 — mirror the originals
rsync -a --delete /mnt/photos/immich-library/ "$DEST/library/"

# 3 — keep the last 14 database dumps
ls -t "$DEST"/immich-db-*.sql.gz | tail -n +15 | xargs -r rm
⌘ssh — raspberrypi
# make it executable, then run it nightly at 03:00
chmod +x ~/immich-backup.sh
(crontab -l 2>/dev/null; echo "0 3 * * * $HOME/immich-backup.sh") | crontab -

Aim for 3-2-1

  • 3 copies: the phone (temporarily), the server, the backup disk.
  • 2 different media: server SSD + external backup drive.
  • 1 offsite: the piece most people skip. Point restic at the same folders and push an encrypted copy to Backblaze B2 or any S3 bucket — pennies per month for photos, and it's the copy that survives fire, theft, or a power surge taking out both local disks.
▎!

A backup you've never restored is a hope, not a backup. Once a quarter, pick one database dump and a random folder of originals and confirm they open. The Immich docs have a full restore procedure ↗ — read it once before you need it.

10

Is it still working?

Self-hosted backup rarely fails loudly. The app silently logs out, iOS stops waking it in the background, and you find out three months later. Two minutes a month — plus one tiny watchdog — closes that gap.

The monthly two-minute check

  1. Open the Immich app and tap the cloud icon: it should show 0 remaining and a recent last-backup time. If it asks you to log in again, you've just found your silent failure.
  2. Compare the counts. iPhone Photos → Library shows your total photos and videos; Immich's Administration → Server stats shows what the server holds. They drift a little (hidden and shared items), but they should track each other.
  3. Take a test photo, open the app for ten seconds, and watch it appear in the web timeline.
  4. Glance at the backup disk: the newest immich-db-*.sql.gz in /mnt/backup/immich should be dated last night.

A dead-man's switch with ntfy

Even better, let the Pi tattle on itself. ntfy.sh delivers free push notifications: install the ntfy app on your iPhone, subscribe to a topic name only you know, and run this check weekly:

shimmich-watchdog.sh
#!/bin/bash
# ping my phone if no new photo has landed in 7 days
NTFY_TOPIC="photos-watchdog-pick-something-unguessable"

RECENT=$(find /mnt/photos/immich-library -type f -mtime -7 -print -quit)
if [ -z "$RECENT" ]; then
  curl -s -d "⚠ No new photos reached the Pi in 7 days — open the Immich app and check it's still logged in." \
    "ntfy.sh/$NTFY_TOPIC"
fi
⌘ssh — raspberrypi
# run it every Monday morning
chmod +x ~/immich-watchdog.sh
(crontab -l 2>/dev/null; echo "0 9 * * 1 $HOME/immich-watchdog.sh") | crontab -
▎i

Two honest caveats: topics on the free public ntfy server are open to anyone who guesses the name, so make yours long and random. And a genuinely photo-free week will trigger it — treat the ping as "go check", not "it's broken".

11

Updates & upkeep

Immich releases often, and occasionally a release changes something you must act on. Ten minutes a month keeps everything healthy.

⌘ssh — raspberrypi
cd ~/immich-app

# 1 — READ THE RELEASE NOTES FIRST:
#     github.com/immich-app/immich/releases — breaking changes are flagged

# 2 — pull new images and restart
docker compose pull
docker compose up -d

# 3 — reclaim space from old images
docker image prune -f
  • Don't auto-update. Skip Watchtower and similar tools for Immich — an unattended breaking release at 3 a.m. is how you lose a weekend. Update deliberately, notes first.
  • Update the phone app around the same time. Server and mobile app are versioned together; drifting far apart eventually causes sync errors.
  • Take a backup before major upgrades. Your nightly script already does this — just check the latest dump exists before you pull.
  • Glance at the dashboard. Administration → Server stats shows library size and job queues; the storage graph tells you when to plan a bigger disk long before it's urgent.
12

Troubleshooting

err / unreachable

App can't reach server

Check the URL is http:// (not https) with port :2283. Same Wi-Fi? Tailscale toggled on? If the server runs a firewall: sudo ufw allow 2283.

err / stalled

Backup stuck partway

Almost always iCloud "Optimise Storage" — originals must download from Apple before uploading. Plug in, stay on Wi-Fi, keep the app foregrounded, be patient.

err / lazy-bg

Background sync is slow

Expected — iOS schedules it opportunistically. Verify Background App Refresh is on and Low Power Mode is off; opening the app uploads instantly.

err / version

Errors right after update

Mobile app and server version have drifted. Update both, restart the app. Check release notes for flagged breaking changes.

err / oom

ML container crashing

Face recognition is the heaviest thing the Pi does. Reduce job concurrency in Administration → Settings → Jobs, or disable smart search — sync works fine without it.

err / power

SSD dropouts, random reboots

Classic underpowered-Pi symptoms. Use the official 27 W PSU, skip unpowered USB hubs, and check dmesg for undervoltage warnings.

err / disk

Disk filling faster than expected

Thumbnails and video transcodes live alongside originals. Check Server stats, prune old docker images, and remember the 2× sizing rule from section 03.

Still stuck? docker compose logs immich_server --tail 100 usually names the problem outright, and the Immich Discord is unusually helpful.

13

Where to go next

Your phone now backs itself up to hardware you own, from anywhere, with real backups behind it. Some good upgrades from here:

  • Family phones. Give each person their own Immich account and put Tailscale on their phones — one server, everyone's camera roll protected, shared albums between accounts.
  • Import your history. Use immich-go to bulk-import a Google Takeout archive or old photo folders, keeping timestamps and albums.
  • External libraries. Have decades of photos already organised in folders? Immich can index them in place without copying or renaming anything.
  • Put it on more screens. The web app already works from any laptop over Tailscale, and Immich Kiosk turns a spare tablet or a TV browser into a rotating photo frame fed straight from your albums.
  • Let the ML soak. After the first big sync, face recognition and smart search jobs churn for a while — on a Pi, give them a few days. When they finish, try searching "birthday cake" or naming the people it found.
  • Reconsider iCloud. Once you've verified a month of clean syncs and restores, you can downgrade the iCloud plan — or keep a small tier as one more redundant copy. Your call, because now it actually is.
