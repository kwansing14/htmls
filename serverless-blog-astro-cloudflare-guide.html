01

Why Astro + Cloudflare

The goal is a blog that is fast for readers, pleasant to write in, and free to run. Most "free" blog stacks fail one of those — they're either a static folder with no way to publish from your phone, or a CMS that bills you the moment you get traffic. This stack threads the needle.

Astro is a content-first web framework. It ships zero JavaScript by default, rendering pages to plain HTML, so a reader's article loads instantly. But Astro is also a real server framework: with an adapter it can render pages on demand, which is exactly what an admin panel needs.

Cloudflare runs your code on its global edge network. The free tier is unusually generous — enough requests, storage, and database reads to run a personal blog comfortably without ever reaching for a credit card.

  • No third-party CMS — no Contentful, no WordPress, no Ghost subscription. You own the whole thing.
  • One codebase — the public site and the admin live in the same Astro project and deploy together.
  • Edge-native data — Cloudflare D1 (SQLite) for posts and R2 for images, both with free tiers.
  • Global by default — every page is served from the Cloudflare location nearest the reader.
▎i

This guide rebuilds the approach from "Building a Serverless Blog with Astro, Cloudflare, and Zero Monthly Cost." The core idea: static pages for readers, a server-rendered admin panel for you — both shipped as a single Astro app on the edge.

02

The architecture in one picture

The whole blog is a single Astro project deployed to Cloudflare. The trick is that not every route is treated the same way. Reader-facing routes are pre-rendered to static HTML; the admin routes are rendered on demand by a Worker.

layer / read

Static reader site

Home, post pages, tags, RSS — pre-built HTML, served from cache at the edge. Instant, JS-free.

layer / write

SSR admin panel

A protected /admin area that runs server-side to create, edit and publish posts.

layer / data

D1 + R2

Cloudflare D1 (SQLite at the edge) stores posts; R2 object storage holds uploaded images.

layer / run

Cloudflare Workers

The Astro adapter compiles your app to a Worker that runs in ~330 cities worldwide.

Because both halves live in one project, publishing a post and shipping a design change use the same deploy. There is no separate backend service to keep alive, and nothing to patch.

▎i

Astro calls this hybrid rendering. The default output is server, and any page you mark export const prerender = true is baked to static HTML at build time. Reader pages opt into static; admin pages stay dynamic.

03

What you need first

Three things, all free:

  • Node.js 18+ and a package manager (npm, pnpm, or bun).
  • A Cloudflare account — the free plan is all this guide uses.
  • The Wrangler CLI, Cloudflare's deploy + resource tool.
⌘Terminal
# check your Node version (need 18 or newer)
node --version

# install Wrangler and log in to Cloudflare
npm install -g wrangler
wrangler login        # opens the browser to authorize
▎!

You do not need a paid plan, a domain, or a credit card to follow along. A custom domain is a nice finishing touch (Section 10) but Cloudflare gives every project a free *.pages.dev / *.workers.dev URL to start.

04

Scaffold the Astro project

Start from Astro's blog template — it ships with content collections, a layout, and example posts so you have a working reader site in seconds.

⌘Terminal
npm create astro@latest my-blog -- --template blog
cd my-blog
npm run dev        # → http://localhost:4321

Open the local URL and you'll see a complete blog. The structure that matters:

📁my-blog/
src/
  pages/            # routes — each file is a URL
    index.astro     # home (static)
    blog/[...slug]  # post pages (static)
    admin/          # we'll add this — SSR
  content/          # content collections (markdown)
  layouts/
  components/
astro.config.mjs    # adapter + output mode live here
▎i

Prefer to write in plain Markdown files and never touch a database? You can stop after Section 6 — that alone is a fast, free, statically-hosted blog. The admin panel and D1 (Sections 7–9) are for when you want to publish without a git commit.

05

Add the Cloudflare adapter

The adapter is what teaches Astro to build for Cloudflare's runtime. Add it and switch the project to server output so individual pages can choose static or on-demand rendering.

⌘Terminal
npx astro add cloudflare
# installs @astrojs/cloudflare and patches astro.config.mjs

Confirm your config sets output: 'server' with the Cloudflare adapter:

jsastro.config.mjs
import { defineConfig } from 'astro/config';
import cloudflare from '@astrojs/cloudflare';

export default defineConfig({
  output: 'server',                 // dynamic by default…
  adapter: cloudflare({
    platformProxy: { enabled: true }, // D1/R2 work in `astro dev`
  }),
});

Now make the reader pages static again — they don't need a server. Add one line to each public page (or the layout they share):

astrosrc/pages/index.astro
---
export const prerender = true;   // bake this page to HTML at build
// …your existing frontmatter…
---
▎i

Default to static, opt into dynamic. A clean rule of thumb: every reader route gets prerender = true; anything under /admin or /api stays server-rendered. Static pages cost you nothing to serve and can't be slow.

06

The reader site

The blog template already renders posts from a content collection — type-safe Markdown defined in src/content.config.ts. Each post page is generated at build time, so readers get pre-rendered HTML straight from cache.

astrosrc/pages/blog/[...slug].astro
---
import { getCollection } from 'astro:content';
import BlogPost from '../../layouts/BlogPost.astro';

export const prerender = true;

// one static route per post
export async function getStaticPaths() {
  const posts = await getCollection('blog');
  return posts.map((post) => ({
    params: { slug: post.id },
    props: post,
  }));
}

const post = Astro.props;
const { Content } = await post.render();
---
<BlogPost {...post.data}>
  <Content />
</BlogPost>

That's a complete, free blog already: write Markdown, commit, deploy. The next sections add the ability to publish from a browser instead of from git.

07

Store posts in D1

To write posts without committing files, you need somewhere to put them. Cloudflare D1 is a serverless SQLite database that lives at the edge — perfect for a blog's modest data, and free up to generous limits.

Create the database

⌘Terminal
wrangler d1 create blog-db
# copy the database_id it prints into wrangler.toml

Bind it so both the deployed Worker and local astro dev can reach it:

tomlwrangler.toml
name = "my-blog"
compatibility_date = "2026-06-01"

[[d1_databases]]
binding = "DB"                 # available as env.DB
database_name = "blog-db"
database_id = "<paste-id-here>"

Define the schema

sqlschema.sql
CREATE TABLE IF NOT EXISTS posts (
  id         INTEGER PRIMARY KEY AUTOINCREMENT,
  slug       TEXT UNIQUE NOT NULL,
  title      TEXT NOT NULL,
  body       TEXT NOT NULL,        -- markdown
  published  INTEGER DEFAULT 0,      -- 0 draft / 1 live
  created_at TEXT DEFAULT (datetime('now'))
);
⌘Terminal
# apply locally for dev, then to the live db
wrangler d1 execute blog-db --local --file=./schema.sql
wrangler d1 execute blog-db --remote --file=./schema.sql
▎i

Reach the binding inside any SSR page or endpoint via Astro.locals.runtime.env.DB. With platformProxy enabled, the same code talks to a local SQLite file during astro dev and to real D1 in production.

08

The server-rendered admin panel

This is the half of the app that runs on demand. A small set of /admin routes reads and writes D1, so you can draft, edit, and publish from any browser — no rebuild required.

A form to create a post

astrosrc/pages/admin/new.astro
---
// SSR (no prerender) — runs on every request
if (Astro.request.method === 'POST') {
  const form = await Astro.request.formData();
  const db = Astro.locals.runtime.env.DB;

  await db
    .prepare('INSERT INTO posts (slug, title, body, published) VALUES (?, ?, ?, ?)')
    .bind(
      String(form.get('slug')),
      String(form.get('title')),
      String(form.get('body')),
      form.get('publish') ? 1 : 0,
    )
    .run();

  return Astro.redirect('/admin');
}
---
<form method="POST">
  <input name="title" placeholder="Title" required />
  <input name="slug"  placeholder="my-first-post" required />
  <textarea name="body" placeholder="Write markdown…"></textarea>
  <label><input type="checkbox" name="publish" /> Publish</label>
  <button>Save</button>
</form>

Render published posts on the reader side by querying D1 from a dynamic post route — or, if you want them cached as static HTML, read D1 in getStaticPaths at build time and redeploy on publish via a deploy hook.

▎!

Sanitize and validate input. Always use bound parameters (.bind(...)) as above — never string-concatenate SQL. Escape or sanitize any HTML you render from post bodies if you allow raw HTML in Markdown.

09

Lock down the admin

The reader site is public; the admin must not be. The simplest, zero-code option on this stack is Cloudflare Access — put a login wall in front of /admin/* at the network edge, before requests ever reach your app.

  • In the Cloudflare dashboard, open Zero Trust → Access → Applications.
  • Add a self-hosted app for the path your-blog.example.com/admin.
  • Create a policy that allows only your email (one-time PIN or Google login).

Access is free for up to 50 users — far more than a personal blog needs. Prefer to handle auth in code instead? A signed, http-only session cookie checked in middleware works too:

tssrc/middleware.ts
import { defineMiddleware } from 'astro:middleware';

export const onRequest = defineMiddleware(async (ctx, next) => {
  if (ctx.url.pathname.startsWith('/admin')) {
    const session = ctx.cookies.get('session')?.value;
    if (!isValid(session)) return ctx.redirect('/login');
  }
  return next();
});
▎i

Cloudflare Access is the recommended path here — it's free, requires no auth code to maintain, and the protected routes never even execute for an unauthenticated visitor. Less code is less to get wrong.

10

Deploy to the edge

One command builds your Astro app and pushes it to Cloudflare's network. Within seconds it's live in hundreds of cities.

⌘Terminal
npm run build          # static pages + the Worker bundle
wrangler deploy        # → https://my-blog.<you>.workers.dev

Continuous deploys from Git

Connect the repo in the Cloudflare dashboard (Workers & Pages → Create → connect to Git) and every push to main builds and ships automatically — no CI to configure.

A custom domain (still free)

  • Add your domain to Cloudflare (free plan) and point the nameservers.
  • In the project's settings, add a custom domain route — DNS and TLS are handled for you.
  • Don't own a domain? The *.workers.dev URL is permanent and free.
▎i

Static reader pages are cached at the edge automatically. For SSR routes you control freshness with Cache-Control headers — but the admin should stay uncached so you always see live data.

11

The zero-dollar math

Here's why the monthly bill is genuinely $0 for a personal blog. Each piece of the stack has a free tier that comfortably covers normal traffic.

ServiceWhat it doesFree tier (typical)
WorkersRuns the app100,000 requests / day
Static assetsReader HTML/CSS/JSUnlimited, served from cache
D1Posts database5 GB · 5M reads / day
R2Image storage10 GB · zero egress fees
AccessAdmin login wallUp to 50 users
DNS + TLSDomain & HTTPSIncluded free

A blog that serves thousands of readers a day sits well inside every one of these limits. You only start paying if the project grows into something much larger — at which point the same architecture scales up without a rewrite.

▎!

Limits drift. Cloudflare adjusts free-tier allowances over time. Treat the numbers above as ballpark and check the current limits on Cloudflare's pricing pages before you rely on them.

12

Where to go next

You now have a serverless blog: static for readers, server-rendered for you, all on the edge for nothing a month. From here:

  • Add an R2 bucket and an upload route so the admin can attach images.
  • Generate an RSS feed and a sitemap (@astrojs/rss, @astrojs/sitemap) — both pre-rendered.
  • Render Markdown post bodies from D1 with a library like marked or Astro's own pipeline.
  • Wire a deploy hook so hitting "Publish" triggers a rebuild and re-caches the static pages.
  • Add full-text search over posts with D1 — still inside the free tier.
